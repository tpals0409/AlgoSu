## When You Let AI Write Code, Who Verifies It?

Ask ChatGPT to write code and you'll get a pretty convincing result. But who checks whether that code conflicts with the existing architecture, violates security rules, or whether the DB migration is rollback-safe?

Uncontrolled AI is dangerous. It generates code quickly, but there's no guarantee that code understands the project's context. If the "just make it work" attitude applies to AI as well, technical debt accumulates faster than when humans write the code.

In the AlgoSu project, we solved this with an **orchestration structure**. AI writes the code, but other AI agents verify it, and an arbiter coordinates the whole operation. This post documents how that system was designed and how it actually works in practice.

## Oracle's Role — The Center of Every Decision

Oracle is the arbiter of AlgoSu's agent system. It is the only agent that communicates directly with the PM (me), and the remaining 11 agents receive tasks and report results exclusively through Oracle.

Oracle's decision-making principle is clear: **Service Stability > Development Velocity > Feature Completeness**.

This priority order is not a mere slogan. It's the actual criteria Oracle uses when resources are scarce or when agents disagree. For example, if Herald requests "I need a new UI component" while Gatekeeper simultaneously reports "There's a vulnerability in JWT validation" — Oracle immediately prioritizes Gatekeeper's security issue.

What Oracle does *not* do is equally important. The actual agent file states this explicitly:

> **Prohibited**: Writing code directly or dictating implementation approaches. Making decisions that undermine core principles (own DB as SSoT, Database per Service, Saga Orchestration). Taking sides with any particular agent.

Oracle decides "what to do," not "how to do it." How to implement Saga logic is Conductor's call, and how to write k8s manifests is Architect's decision.

### Tier 1 — Mission Critical

When the service breaks, everything stops. Tier 1 agents are the ones who prevent that "breaking."

**Conductor (Orchestrator)** directs the entire lifecycle of code submissions. It governs the Saga Orchestrator's state transitions (`DB_SAVED -> GITHUB_QUEUED -> AI_QUEUED -> DONE`) and never reverses the idempotency rule ("DB first, MQ second") under any circumstances. Conductor's agent file even specifies the pre-submission validation order: (1) membership check, (2) GitHub connection check, (3) deadline check, (4) duplicate submission idempotency check.

**Gatekeeper (Guardian)** is the sole entry point for all external requests. The agent file states it unequivocally: "Requests you have not authorized cannot reach internal services." Security always takes priority over convenience, and suspicious requests are never permitted.

**Librarian (Records Keeper)** is the most careful agent when it comes to protecting data. The opening line of the agent file is striking: "Data, once stored incorrectly, is hard to recover. You are the agent that must be the most careful in this entire project." Migrations without `down()` are outright prohibited, and the Expand-Contract pattern (3-phase deployment) is enforced.

### Tier 2 — Core

These agents handle infrastructure foundations and core business logic.

**Architect (Infrastructure Designer)** manages k3s clusters, CI/CD pipelines, and monitoring. Rejects deployment requests with the `latest` tag and never allows Pods without Resource Limits — no exceptions. **Postman (Delivery Agent)** handles GitHub Push while always remembering it's an "auxiliary service" — "Even if delivery fails, the submission itself has already succeeded." **Curator (Problem Setter)** controls problem management and deadlines, and **Scribe (Chronicler)** is solely responsible for the accuracy and currency of all project memory and documentation.

### Tier 3 — Enhancement

These roles enhance service value. They work after Tiers 1 and 2 are stable.

**Sensei (Analyst)** calls the Claude API to analyze code. When the Circuit Breaker is OPEN, API calls are absolutely prohibited. **Herald (Messenger)** implements frontend pages but must always use Palette's design tokens. **Palette (Designer)** owns the design system and prohibits inline hardcoding like `bg-[#...]`. **Scout (Recon)** is the only agent that doesn't write code — it experiences the service through a user's eyes and only reports feedback.

## persona-base.md — The SSoT for Shared Rules

For 12 agents to behave consistently, shared rules are necessary. In AlgoSu, a single file called `agents/_shared/persona-base.md` serves this role. It is the SSoT (Single Source of Truth).

Every agent must read this file before starting work. Each agent file contains the same directive:

> **Shared Rules**: Reference: `agents/_shared/persona-base.md` (mandatory read before starting)

Let's look at the key rules defined in this file.

### Reporting Chain

<Callout type="info" title="Reporting Chain (persona-base.md)">
- Only Oracle communicates directly with the PM. Agents are strictly prohibited from responding to the PM directly
- All work is performed under Oracle's explicit approval. Independent resource access is forbidden
- Report to Oracle after task completion. Do not exceed granted authority
</Callout>

There's no scenario where an agent says "I thought this would be good, so I went ahead and did it." Nothing happens without Oracle's approval.

### Escalation Principle

<Callout type="warn" title="Escalation Principle">
- When judgment exceeds scope or conflicts arise between agents → Escalate to Oracle (Arbiter)
- If a decision can't be made within 4 hours → Report to PM
</Callout>

Agents don't know everything either. When they don't, they escalate rather than making a rough guess. The 4-hour timeout prevents infinite loops of disagreement between agents. In practice, when Palette and Herald disagree on a component spec, Oracle mediates.

### Interface Contract Principle

<Callout type="info" title="Interface Contract Principle">
- Notify related agents 24 hours before changes
- Prioritize backward compatibility; breaking changes require Oracle approval
- All inter-service communication must include the X-Internal-Key header
</Callout>

This is the rule requiring advance notice when changing API specs. It prevents one service's API changes from breaking another service in a microservices environment.

### Code Writing Rules

<Callout type="success" title="Code Writing Rules (All Agents)">
- Clean code: meaningful naming, single responsibility per function, 20 lines max, DRY
- SOLID principles: SRP, OCP, LSP, ISP, DIP
- Mandatory comments: file header (`@file`, `@domain`, `@layer`, `@related`) + function JSDoc
- No inline hardcoding: `bg-[#...]` prohibited, use Tailwind token classes
- JSON structured logging
</Callout>

These rules apply identically to all 12 agents. Whether code is written by Conductor or Herald, it follows the same style and the same quality standards. Modifying `persona-base.md` changes the behavior of all agents at once.

### Workflow

The standard workflow defined in `persona-base.md` looks like this:

## Agent File Structure — Separating Roles from Tools

The file structure of the agent system reveals the design intent more clearly.

```text
agents/
├── _shared/
│   └── persona-base.md          # Shared rules SSoT (mandatory read for all agents)
│
└── commands/
    ├── conductor.md             # Conductor role/responsibilities/judgment criteria
    ├── gatekeeper.md            # Gatekeeper role/responsibilities/judgment criteria
    ├── librarian.md             # Librarian role/responsibilities/judgment criteria
    ├── architect.md             # Architect role/responsibilities/judgment criteria
    ├── postman.md               # Postman role/responsibilities/judgment criteria
    ├── curator.md               # Curator role/responsibilities/judgment criteria
    ├── scribe.md                # Scribe role/responsibilities/judgment criteria
    ├── sensei.md                # Sensei role/responsibilities/judgment criteria
    ├── herald.md                # Herald role/responsibilities/judgment criteria
    ├── palette.md               # Palette role/responsibilities/judgment criteria
    ├── scout.md                 # Scout role/responsibilities/judgment criteria
    │
    ├── annotate.md              # Tool: annotation rules
    ├── monitor.md               # Tool: monitoring rules
    ├── cicd.md                  # Tool: CI/CD rules
    ├── migrate.md               # Tool: migration rules
    ├── conventions.md           # Tool: code conventions
    ├── ui.md                    # Tool: UI design system
    └── adr-sprint.md            # Tool: Sprint ADR template
```

**Agent files** and **tool files** are separated. Agent files (`conductor.md`, `gatekeeper.md`, etc.) define roles, responsibilities, collaboration interfaces, judgment criteria, and escalation conditions. Tool files (`annotate.md`, `migrate.md`, etc.) contain specific rules and procedures.

Each agent references only the tools it needs. For example, Librarian's agent file specifies:

<Callout type="info" title="Librarian Tool References (from librarian.md)">
- Migration (required reading): `agents/commands/migrate.md`
- Annotations: `agents/commands/annotate.md`
- Monitoring: `agents/commands/monitor.md`
- CI/CD: `agents/commands/cicd.md`
</Callout>

Meanwhile, Scout (Recon) doesn't write code, so its tool references are limited to just `ui.md`. This separation ensures that agents aren't confused by irrelevant rules.

## Slash Command System — The PM's Interface

The interface through which the PM (me) communicates with the agent system consists of 5 slash commands. All are defined in the `.claude/commands/` directory.

### /start — Session Start

The command that kicks off a sprint. Internally, here's what happens:

1. **Context loading**: Reads `sprint-window.md` (sliding window) to understand the previous sprint's state
2. **State validation**: Checks if the previous sprint terminated normally. Shows a warning if still `active`
3. **Sprint title decision**: Passed as an argument, or kept from the previous title, or asks the user
4. **Sprint activation**: Changes `sprint-window.md` status to `active` and records the current HEAD commit
5. **Dashboard output**: Shows current/previous sprint status, remaining items, and work guidance

The dashboard output always ends with this guidance:

> Request all work through `/algosu-oracle`.
> Oracle will delegate appropriately to the 12 specialized agents.

### /stop — Session End

The command that wraps up a sprint. It consists of 6 stages, each designed to be recoverable even if it fails.

1. **Check current state**: Read `sprint-window.md` + run `git log`. Asks whether to cancel if there are no commits
2. **Auto-draft + user confirmation**: Auto-generates a work summary based on git log, gives the PM a chance to edit
3. **Sprint ADR creation**: Creates `docs/adr/sprints/sprint-{N}.md`. Records decisions, patterns, and lessons
4. **Sliding window update**: Promotes the completed sprint to `[1] Complete` and writes a new `[2] Next` section
5. **MEMORY.md update + consistency verification**: Table update, 3-file consistency check, transitions to `idle` only after all verifications pass
6. **Completion report**: Summarizes the close-out results for the PM

The core design principle is "state transitions happen only after all updates are complete." Because the transition to `status: idle` happens last, if interrupted midway, re-running `/stop` resumes from the missing step. It's a similar concept to the Saga pattern.

### /algosu-oracle — Task Request

The core command for requesting work from Oracle. When Oracle receives a request, it works like this:

1. Reads `agents/_shared/persona-base.md` to review shared rules
2. Analyzes the task content and identifies required agents
3. Reads the relevant agent file (`agents/commands/{name}.md`) to understand the role
4. Assigns the task to agents and aggregates results
5. Reports results to the PM

The agent reference paths are specified in Oracle's file:

<Callout type="info" title="Oracle Agent Reference Paths (from oracle.md)">
When assigning tasks, read the relevant agent file to fulfill the role:
`agents/commands/{name}.md` — conductor, gatekeeper, librarian, architect, scribe, postman, curator, herald, palette, scout, sensei
</Callout>

### /algosu-review — Code Review

A dedicated code review command. It verifies changes through common checklists, security checklists, and migration checklists. Approval criteria vary by PR type.

| Type | Approvals | Required |
|------|-----------|----------|
| General | 1 | CODEOWNERS |
| Security | 2 | Gatekeeper |
| Infrastructure | 1 | Architect |
| DB Schema | 2 | Architect + Conductor |
| Breaking | 2 | Oracle (final) |

Security-related changes always require Gatekeeper review, and DB schema changes need approval from both Architect and Conductor. Oracle has final authority only for Breaking Changes.

### /algosu-adr-blog — Convert ADRs to Blog Posts

A command that converts formal ADRs or Sprint ADRs into tech blog MDX posts. It transforms the technical report tone of ADRs into "easy-to-read blog posts" while preserving code blocks and diagrams. This very post is a product of that system.

Each agent has different escalation conditions. Conductor escalates decisions that affect business rules, like "Saga compensation transaction logic changes" or "deadline policy changes." Sensei escalates cost-related judgments, like "Claude API estimated cost threshold exceeded."

Scout is unique. As an agent that doesn't write code, it still escalates when it discovers "service unreachable" or "security vulnerabilities (personal data exposure)." Because it only views the system from a user's perspective, it catches problems developers miss.

## Real Sprint Orchestration

For a concrete example, let's look at Sprints 63–64 — where we added Discord feedback notifications and fixed bugs.

**Sprint 63 — Discord Feedback Notifications (2 commits, 10 files)**

```text
PM: "When an admin leaves feedback, send a Discord notification"

Oracle task analysis:
  - This is a new feature addition
  - Discord webhook logic needed in Identity service (Conductor's domain? No, Identity)
  - Notification trigger addition needed in Gateway (Gatekeeper's domain)
  - Tests needed

Oracle assignment:
  1. Gatekeeper → Webhook trigger API on feedback creation
  2. Conductor → Discord webhook logic in Identity service
  3. Scribe → Sprint ADR recording
```

**Sprint 64 — Discord Notification Not Sending Bug Fix (3 commits, 8 files)**

In the very next sprint, a "notifications aren't being sent" bug was discovered. This is where Oracle's priority principle kicks in — service stability comes first. New feature development is paused and the bug fix proceeds first. Additionally, the missed case of "notification on feedback resolution" was handled together.

This is how work progresses sprint by sprint. Every sprint, the `/stop` command generates a Sprint ADR, and history accumulates in `MEMORY.md`. Sixty-seven sprints repeated this cycle.

## Agent Model Assignment — The Right Model for the Role

An interesting point is that not all agents use the same AI model. Each agent file's frontmatter specifies its model.

| Model | Agents | Reason |
|-------|--------|--------|
| `claude-opus-4-6` | Oracle, Conductor, Gatekeeper, Librarian | Complex judgment, security decisions, data integrity |
| `claude-sonnet-4-6` | Architect, Postman, Curator, Scribe, Sensei, Herald, Palette, Scout | Implementation tasks, cost efficiency |

Tier 1 Mission Critical agents use the more powerful model (Opus). Planning decisions, security judgments, and data schema design are tasks where mistakes are costly. Tier 2–3 agents are well-served by Sonnet — implementation work just needs to follow the rules.

## Strengths of This Structure

**1. Consistent Code Quality**

Over 67 sprints, code style never wavered. That's because `persona-base.md` operates as SSoT. Annotation rules, naming conventions, and error handling patterns remain identical from start to finish.

**2. Context Retention**

The triple structure of Sprint ADR + `MEMORY.md` + `sprint-window.md` preserves context. Decisions from three months ago can be found in five seconds. Since Scribe manages memory within 200 lines, only the latest context is ever quickly accessible.

**3. Safe Changes**

Librarian enforces 3-phase Expand-Contract for DB changes, Gatekeeper verifies security rules, and Architect checks resource limits. One agent's changes are structurally prevented from breaking another domain.

**4. Extensible Architecture**

When a new role is needed, just add an agent file to `agents/commands/`. Shared rules are automatically applied by reading `persona-base.md`.

## Limitations of This Structure

Let's be honest about the limitations too.

**1. Context Window Cost**

Every time an agent starts work, it reads `persona-base.md`, its own agent file, and tool files. This "pre-loading" consumes context window. For complex tasks, there may be less room left for reading actual code.

**2. Oracle Bottleneck**

Since all work goes through Oracle, Oracle can become a bottleneck. While it's possible to assign tasks to multiple agents simultaneously, final verification is processed sequentially by Oracle.

**3. Rule Maintenance Cost**

Keeping agent files, tool files, and shared rules up to date is also a cost. As the project evolves, rules must evolve with it. Scribe handles this role, but the final judgment is still the PM's (my) responsibility.

**4. Lack of Domain Intuition**

"Does the user actually need this feature?" is a question agents can't answer. Technical execution can be delegated to agents, but directional decisions must be made by humans.

## Closing — Controlled AI Is Useful AI

Letting AI write code isn't the hard part. The hard part is making that AI follow the project's rules, avoid conflicts with other domains, and be reversible when mistakes happen.

AlgoSu's orchestration structure is one answer to this problem. It's not a perfect answer, but it's the structure that made it possible to complete 67 sprints at consistent quality. Oracle decides, 11 agents execute, and `persona-base.md` maintains consistency. Uncontrolled AI is dangerous, but controlled AI follows rules better than human teammates.
