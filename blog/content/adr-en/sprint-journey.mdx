## 67 Sprints, 2,432 Tests

Let's start with the numbers.

- **67** sprints
- **2,432** test cases
- **6** microservices + **1** frontend
- **3** databases (identity_db, problem_db, submission_db)
- **15** CI jobs
- **12** AI agents
- **34** Identity API endpoints (0 → 34 in Sprint 51)
- **Identity branches coverage** 99.51%

From Sprint 3, when AlgoSu began, to Sprint 67, when I could say it's "approximately 99% complete," I went through four major turning points. This post is a record of that journey.

In the beginning, it was naturally "just make it work." Problem CRUD, Submission CRUD, user authentication. The most fundamental things.

### The OAuth Struggle

I integrated all three OAuth providers at once: Google, Naver, and Kakao. Each platform's callback URL format was different, the returned profile data structures were different, and the token refresh policies were different. "Integrating OAuth is easy" is only true when you're integrating one.

But one decision I made at this point proved enormously helpful later — **httpOnly Cookie JWT**. Choosing httpOnly cookies over localStorage to reduce the XSS attack surface seemed like a small choice at the time. But when Sprint 65 came around and I had to mass-delete 11 localStorage token functions, I thought, "Good call on that early decision."

### Database per Service

Around Sprint 12, I separated the databases into three: identity_db, problem_db, submission_db. Honestly, at that point it felt like overkill. There were only three services — why three databases?

But when Sprint 51's Gateway → Identity DB separation came around, having the databases already split made the migration far smoother. The "excessive" early design ended up saving my future self.

### Phase 1 Lesson
> **"To reduce future work, you need to do a little more now."**
> DB separation, httpOnly Cookie, external IDs (publicId) — decisions where I invested slightly more time upfront saved cumulative hours across dozens of subsequent sprints.

Phase 2 was the journey from "microservices that act like a monolith" to "real microservices." The services were physically separated, but in practice they were tightly coupled through synchronous HTTP calls.

### Saga Orchestrator

The code submission workflow grew complex. When a user submits code:

1. Submission service saves the submission
2. GitHub Worker pushes to GitHub
3. AI Analysis analyzes the code
4. Results must flow back to Submission

What if any of these fails? A partially completed state can't be left behind. We adopted the Saga Orchestrator pattern so the Submission service manages the state of the entire workflow.

### RabbitMQ Introduction

Implementing the Saga made it clear we needed asynchronous inter-service communication. We introduced RabbitMQ and transitioned to an event-driven architecture. The Submission → RabbitMQ → GitHub-Worker (Push) / AI-Analysis (analysis) flow was born in this phase.

### AI Analysis and Circuit Breaker

The AI Analysis service calls the Claude API. External APIs can slow down or fail at any time. We added a Circuit Breaker so that consecutive Claude API failures would block requests for a period. This was built as a separate service with FastAPI + Python — a tradeoff that broke NestJS uniformity in exchange for the benefits of the Python ML ecosystem.

### Phase 2 Lesson
> **"Async is 10x harder to debug."**
> With synchronous calls, errors blow up immediately. With async event-driven systems, tracking down "the message was published, so why isn't it being consumed?" was truly painful. But without async, we couldn't decouple the services.

Phase 3 was the journey from "code that runs" to "an operable system." And this is where the biggest event occurred.

### The Birth of CI 15 Jobs

Initially, CI had 3 jobs: lint, test, build. Since AI generates code quickly, I figured CI could be simple too.

**Wrong.**

There was an incident where `git commit --amend` followed by a force-push caused `dorny/paths-filter` to miss changes entirely, skipping the build completely. After that, a force-push ban was introduced along with progressive CI hardening, eventually growing to 15 jobs.

From secret-scan, detect-changes, quality checks (NestJS/Python/Frontend), tests (Jest/pytest/Vitest), audit, build, to GitOps sync. The lesson: the faster AI generates code, the tighter the verification pipeline needs to be.

### ADR-001: Gateway → Identity DB Separation (Sprint 51)

This was a decision significant enough to warrant its own ADR document. The Gateway service's direct access to identity_db violated the Database per Service principle. Six entities (User, Study, StudyMember, StudyInvite, Notification, ShareLink) were being directly CRUDed across 19 files.

The decision was a full migration:
- **34 new API endpoints** built in the Identity service
- All 19 files in Gateway converted to **IdentityClient HTTP calls**
- 4 entity files deleted, common types extracted

Since it's k8s internal network calls, the added latency was about 1ms. Gateway branches coverage held at 97.79%. All of this happened in a single Sprint 51.

### GitOps Automation

We connected ArgoCD with k3s to build a complete GitOps pipeline. `git push` → CI 15 jobs → ghcr.io image build → aether-gitops repo tag update → ArgoCD → k3s deployment. No more human manual intervention at any step.

### Phase 3 Lesson
> **"Time invested in automation always pays off."**
> Building CI 15 jobs took several sprints, but afterward, "Is this commit safe?" never required human judgment again. And having test coverage backing us up made it possible to execute bold refactoring like ADR-001.

Phase 4 was the final stretch from "a working system" to "a finished product." No major architecture changes, but the battle over details continued.

### Sprint 62: Admin Page Lessons

The feedback management page was filtering by status on the client side. Combined with server-side pagination (20 per page), what happens? Only the current page's 20 items get filtered, and feedback with that status on other pages is invisible.

An obvious bug, but one I didn't catch while building it. We switched to server-side filtering and included total counts in the response to fix it. Eight commits, 17 files changed.

Screenshot storage was also switched from JPEG to WebP. Since they're stored as Base64 Data URLs in DB TEXT columns, file size mattered. `canvas.toDataURL('image/webp', 0.65)` yielded a 25–35% reduction compared to JPEG.

### Sprint 63–64: The Discord Notification Trap

Discord feedback notifications were technically simple. Create a DiscordWebhookService in the Identity service, fire a webhook on feedback creation — done. Fire-and-forget pattern, no impact on main logic.

**The trap was in deployment.**

Sprint 63: Code was complete and pushed. Secrets were created, source repo manifests were written. But Discord notifications weren't arriving. The cause? **The aether-gitops repo's env block hadn't been updated.** ArgoCD syncs the GitOps repo's manifests, but we'd only modified the source repo and forgotten about GitOps.

Sprint 64 fixed this while also implementing `FEEDBACK_RESOLVED` in-app notifications. But here we fell into another trap: an enum was added to the Identity entity but not propagated to the Gateway enum or Frontend types. Sprint 66 ended up requiring a full 3-tier sync (Identity → Gateway → Frontend).

### Sprint 65: Security Audit

Despite having switched to httpOnly Cookies long ago, localStorage-related code remained. `setToken`, `getToken`, `isTokenExpired` — **11 functions** removed. They were already unused in production, but test code still referenced them, so those had to be cleaned up together.

Adding a `NODE_ENV` guard to the `DEV_MOCK` bypass also happened in this sprint. If `NEXT_PUBLIC_DEV_MOCK=true` got bundled into a production build, the entire Edge Middleware would be bypassed — a latent risk.

One commit changed 7 files: **+42 lines / -440 lines**. I didn't know a sprint dedicated to deleting code could feel this satisfying.

### Sprint 67: Tech Blog

Removed the sprint pages and replaced them with ADR-based tech posts. And here I am, writing this very post. The blog.yaml GitOps manifest's imagePullPolicy and GHCR configuration were synced at the same time.

### Phase 4 Lesson
> **"Adding one enum changes 5 files."**
> Type synchronization in MSA takes more effort than you'd think. The propagation checklist created in Sprint 66 for NotificationType additions (Identity entity → Gateway types → Frontend api.ts → NotificationBell → NotificationToast) would have prevented Sprint 64's troubles if it had existed from the start.

## Pivotal Sprints

### 1. Sprint 12: Database per Service Confirmed

The moment I convinced myself, "This feels excessive now, but I'll be grateful later." And indeed, Sprint 51's ADR-001 work was made considerably easier by this decision. A gift to my future self.

### 2. Sprint 22: RabbitMQ Event Bus Introduction

The sprint where I felt the limits of synchronous HTTP calls firsthand. The Submission → GitHub Push → AI analysis async pipeline was born here. Debugging got harder, but inter-service coupling dropped dramatically.

### 3. Sprint 51: ADR-001 Gateway → Identity DB Separation

The largest single refactoring in the project. Identity API went from 0 to 34 endpoints, and 19 Gateway files were converted to HTTP clients. This was only possible because test coverage backed us up.

### 4. Sprint 63–64: GitOps Manifest Sync Lesson

Code was perfectly written, but deployment didn't work. The rule "when you modify source repo manifests, you must also update the GitOps repo" was established here. The habit of verifying actual injection via `kubectl exec -- printenv` also started in this sprint.

### 5. Sprint 65: The Joy of Deleting Code

+42 lines / -440 lines. A sprint that cleaned up localStorage token remnants and added DEV_MOCK security guards. I realized that removing unnecessary code makes the system safer than adding new features.

## What Went Well vs. What I'd Do Differently

### What Went Well

**1. Separating DBs from the start.** When the big refactoring came in Phase 3, having databases already separated made the scope of work crystal clear.

**2. Being obsessive about tests.** 2,432 tests and 99.51% branches coverage. Without these, large-scale refactoring like ADR-001 would have been unthinkable.

**3. Writing ADRs.** Recording why each decision was made meant never repeating the same deliberation when similar situations arose later. Sprint ADRs became the project's permanent record.

**4. Building the AI agent system.** With 12 agents each responsible for their own domain, I could focus solely on "what needs to be done." "How to do it" was handled by the agents.

### What I'd Do Differently

**1. Not having TLS yet.** Over 67 sprints, TLS certificate setup kept getting deferred. Feature implementation took priority, but from a security standpoint, it should have come sooner.

**2. No automated MSA type synchronization.** Every time an enum is added, 5 files must be manually updated. A code generator or shared package would have prevented the sync failures in Sprints 64/66.

**3. Connecting monitoring alerts late.** Alertmanager was configured, but the actual notification channel (Discord) wasn't connected until Sprint 63. "Having monitoring tools" and "having monitoring that works" are different problems.

## What's Next?

By Sprint 67, AlgoSu has reached a level where I can call it "approximately 99% complete." But there's still work remaining.

**Non-blocking follow-ups:**
- TLS certificate introduction
- Existing problem DB level=2 data correction (residual AddProblemModal bug)
- Existing user GitHub token reconnection

**Architecture review candidates:**
- Converting other services (submission, problem, github-worker) → direct Identity calls (currently proxied through Gateway)
- Physical separation of postgres instances (OCI ARM resource limitations)
- React Query adoption (currently pure Context + hooks)
- Swagger/OpenAPI documentation

One thing has become certain after 67 sprints: even solo, with the right system in place, you can build a solid service. Twelve agents, 15 CI jobs, and 2,432 tests were my "team."

Sprint 68 is still blank. I don't know what comes next, but I know this journey doesn't end here.
