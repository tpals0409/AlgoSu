## What If Pressing One Button Moves 6 Services?

A user writes their code and hits the "Submit" button. A loading spinner briefly appears on screen, and shortly after, AI analysis results show up. From the user's perspective, that's all there is to it.

But behind the scenes, Gateway validates the JWT, Submission Service saves to the DB, GitHub Worker pushes the code via RabbitMQ, AI Analysis calls the Claude API, and real-time status is delivered via Redis Pub/Sub to SSE. Six services run a relay to complete a single submission.

I designed and operate this architecture entirely on my own. In this post, I'll draw a bird's-eye view of AlgoSu's full architecture and document each service's role, how they communicate, and how they're deployed.

## A Single Submission's Journey — Saga Orchestrator

When a user submits code, the Submission Service's Saga Orchestrator governs the entire flow. Here are the state transitions as defined in the actual code:

Three core design principles are at work:

**1. Idempotency Order: DB First, MQ Second**

When a service restarts, DB records persist but MQ messages are lost. So we always update DB state first, then publish the RabbitMQ message. Reversing the order would risk duplicate publication.

**2. Optimistic Locking Prevents Backward Progression**

Every state transition includes a `WHERE sagaStep = currentStep` condition. If the submission has already moved to the next step, `affected === 0` prevents duplicate processing.

**3. Timeouts + Automatic Resumption**

When the service restarts, `onModuleInit` finds incomplete Sagas within the last hour and automatically resumes them. Even lost messages get recovered.

## Database per Service — 3 Separate Databases

We followed the textbook microservice principle: each service gets its own database.

<ServiceGrid cols={2}>
  <ServiceCard name="PostgreSQL #1 (Shared)" tech="identity_db + submission_db" accent={1}
    role="users, studies, notifications, shares / submissions, code_reviews, study notes" />
  <ServiceCard name="PostgreSQL #2 (Separate)" tech="problem_db" accent={4}
    role="problems, deadlines — physically separate instance" />

`identity_db` and `submission_db` are logically separated within the same PostgreSQL instance, while `problem_db` is physically separated on an entirely different PostgreSQL instance (`postgres-problem`). Running three instances on OCI ARM free tier's limited resources was impractical, so we compromised with two.

All PostgreSQL instances have `statement_timeout 200ms` configured to block slow queries. This prevents a single runaway query from paralyzing the entire service.

## 4 Communication Patterns

Inter-service communication uses 4 patterns depending on the use case.

### 1. Synchronous HTTP + X-Internal-Key

Calls from Gateway to Identity, Submission, and Problem services use HTTP. For security, all internal calls include the `X-Internal-Key` header.

```typescript
// services/gateway/src/common/guards/internal-key.guard.ts

@Injectable()
export class InternalKeyGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest<Request>();
    const providedKey = request.headers['x-internal-key'];
    const expectedKey = this.configService.getOrThrow<string>('INTERNAL_API_KEY');

    if (!providedKey || typeof providedKey !== 'string') {
      throw new UnauthorizedException('Internal API Key is required.');
    }

    if (!this.timingSafeEqual(providedKey, expectedKey)) {
      throw new UnauthorizedException('Invalid Internal API Key.');
    }

    return true;
  }

  private timingSafeEqual(a: string, b: string): boolean {
    const bufA = Buffer.from(a);
    const bufB = Buffer.from(b);
    if (bufA.length !== bufB.length) return false;
    return timingSafeEqual(bufA, bufB);
  }
}
```

`crypto.timingSafeEqual` prevents timing attacks. Keys are managed as k8s Secrets with separate keys per service. Hardcoding is strictly prohibited.

### 2. RabbitMQ Async Messaging

GitHub Push and AI analysis after submission are long-running operations. Since we can't make users wait, they're processed asynchronously via RabbitMQ.

GitHub Worker's `prefetch=2` is intentional — a concurrency limit considering GitHub API's Rate Limit and OCI Free Tier's constrained resources. On failure, messages go through Retry then to the DLQ (Dead Letter Queue).

### 3. Redis Pub/Sub — Real-time Status Propagation

Every time a submission's status changes, a message is published via Redis Pub/Sub. The channel format is `submission:status:{submissionId}`.

### 4. SSE (Server-Sent Events) — Browser Real-time Streaming

Gateway's SSE Controller subscribes to Redis Pub/Sub and delivers status to the browser as `text/event-stream`.

Several safeguards are in place for SSE connections:

- **5-minute max connection time** — prevents infinite connections; client reconnects on timeout
- **Shared Redis subscriber** — creating a Redis instance per connection would exhaust the connection pool, so a single subscriber manages channel listeners
- **30-second heartbeat** — periodically sends `: heartbeat\n\n` to prevent proxies from killing idle connections
- **Ownership verification** — only the submission's owner can subscribe to its status (IDOR defense)

## OAuth + JWT + httpOnly Cookie Authentication

Here's the authentication flow at a glance:

    O-->>G: access_token

    G->>G: JWT issuance (HS256, 1h)
    G-->>B: Set-Cookie (httpOnly, Secure, SameSite)

A few design decisions worth sharing:

**Why httpOnly Cookie**: Storing JWT in localStorage is vulnerable to XSS attacks. httpOnly Cookies are inaccessible via JavaScript, significantly reducing the risk of token theft.

**Auto-renewal**: TokenRefreshInterceptor checks JWT remaining time on every request. If within 5 minutes (300 seconds) of expiry, it issues a new token and includes it in the response cookie. The user's session is seamlessly extended without a separate refresh request.

**Algorithm pinning**: Only `HS256` is allowed. The `algorithms: ['HS256']` option is explicitly set to prevent algorithm confusion attacks.

CI consists of 15 jobs. Path filters ensure only changed services are tested and built. ARM aarch64 images are built and pushed to GHCR with `main-{git-sha}` tags. The `latest` tag is never used.

The first CI job is always `secret-scan`. It detects secret leaks with gitleaks and blocks `.env` file commits. Only after the security scan passes can the build proceed.

### Zero-Downtime Deployment Strategy

All services use the same RollingUpdate strategy:

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0   # Keep existing Pods
    maxSurge: 1          # Add 1 new Pod, then switch
```

With `maxUnavailable: 0`, at least one Pod is always handling requests during deployment. The existing Pod terminates only after the new Pod passes its readinessProbe.

Identity, Submission, and Problem services run DB migrations via initContainer before starting the app:

```yaml
initContainers:
  - name: db-migrate
    command: ["node", "./node_modules/typeorm/cli.js",
              "migration:run", "-d", "dist/src/database/data-source.js"]
```

Total memory: approximately 2.2Gi by request, approximately 5.9Gi by limit. Subtracting OS and k3s overhead from 24GB leaves comfortable margin.

## Security Layers — Preventing Default-Configuration Mistakes

A common security configuration is applied to all containers:

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1001
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop:
      - ALL
```

No running as root, privilege escalation is blocked, and the filesystem is mounted read-only. Only necessary write paths (`/tmp`, `/var/log`) are mounted as emptyDir.

At the network level, we start with a default-deny policy and only allow required inter-service communication via NetworkPolicy.

## AI Analysis — Circuit Breaker and Safe Parsing

The AI Analysis service calls the Claude API (`claude-haiku-4-5-20251001`, `MAX_TOKENS=8192`). Given the external API dependency, we applied the Circuit Breaker pattern.

When the API fails consecutively, the Circuit Breaker transitions to OPEN state and blocks API calls for a period. During this time, incoming requests trigger a `CircuitBreakerOpenError` and are NACK+requeued in RabbitMQ.

Claude's response parsing is also implemented defensively. LLMs don't always return perfect JSON.

## Closing — A Practical MSA for a Small Team

"You're a solo developer — do you really need MSA?" is a question I get occasionally. Honestly, I didn't start with 6 services from day one. They were naturally separated as features grew.

Looking back, there are a few practical lessons:

**Separating DBs early was the right call.** Separating them later leads to migration hell. Following the Database per Service principle from the start keeps inter-service coupling loose.

**Async processing is essential.** If GitHub Push and AI analysis were synchronous, users would wait over 30 seconds. With RabbitMQ + Saga pattern, we respond immediately on submission and deliver progress via SSE in real time.

**Design within free tier constraints.** OCI ARM free tier is generous but not unlimited. GitHub Worker's prefetch=2, AI Analysis's daily limit check, Redis's maxmemory 200MB — all resource-conscious decisions.
