## AI에게 코드를 맡기면, 누가 검증하나요?

ChatGPT에게 코드를 작성해달라고 하면 꽤 그럴듯한 결과물이 나옵니다. 하지만 그 코드가 기존 아키텍처와 충돌하지 않는지, 보안 규칙을 위반하지 않는지, DB 마이그레이션이 롤백 가능한지 — 이런 것들은 누가 확인하나요?

통제 없는 AI는 위험합니다. 빠르게 코드를 생성하지만, 그 코드가 프로젝트의 맥락을 이해하고 있다는 보장이 없습니다. "일단 돌아가면 됐지"라는 태도가 AI에게도 적용되면, 기술 부채는 사람이 작성할 때보다 더 빠르게 쌓입니다.

AlgoSu 프로젝트에서는 이 문제를 **오케스트레이션 구조**로 해결했습니다. AI가 코드를 작성하되, 그 AI를 다른 AI가 검증하고, 전체를 조율하는 심판관이 존재하는 구조입니다. 이 글에서는 그 체계가 어떻게 설계되었고, 실제로 어떻게 동작하는지를 기록합니다.

> 이 글은 [AlgoSu 기술 블로그 시리즈](/posts/agent-orchestration-solo-dev) 6편 중 3편입니다.
> 이전 글: [6개 서비스가 하나의 제출을 처리하기까지](/posts/system-architecture-overview)

## Oracle의 역할 — 모든 결정의 중심

Oracle은 AlgoSu 에이전트 체계의 심판관입니다. PM(저)과 직접 소통하는 유일한 에이전트이고, 나머지 11개 에이전트는 Oracle을 통해서만 작업을 받고 결과를 보고합니다.

Oracle의 의사결정 원칙은 명확합니다. **서비스 안정성 > 개발 속도 > 기능 완성도**.

이 우선순위는 단순한 구호가 아닙니다. 실제로 리소스가 부족하거나 에이전트 간 의견 충돌이 발생할 때, Oracle이 이 기준으로 판단합니다. 예를 들어 Herald가 "새 UI 컴포넌트가 필요해요"라고 요청하고, 동시에 Gatekeeper가 "JWT 검증에 취약점이 있어요"라고 보고하면 — Oracle은 주저 없이 Gatekeeper의 보안 이슈를 먼저 처리합니다.

Oracle이 하지 않는 것도 중요합니다. 실제 에이전트 파일에 이렇게 명시되어 있습니다.

> **금지**: 직접 코드 작성 또는 구현 방식 지시. 핵심 원칙(자체 DB SSoT, Database per Service, Saga Orchestration) 훼손 결정. 특정 Agent의 편을 드는 발언.

Oracle은 "무엇을 할지"를 결정하지, "어떻게 할지"는 각 에이전트의 전문 영역입니다. Saga 로직을 어떻게 구현할지는 Conductor가 판단하고, k8s 매니페스트를 어떻게 작성할지는 Architect가 결정합니다.

### Tier 1 — Mission Critical

서비스가 깨지면 모든 것이 멈춥니다. Tier 1 에이전트들은 그 "깨짐"을 방지하는 역할입니다.

**Conductor(지휘자)** 는 코드 제출의 전체 생명주기를 지휘합니다. Saga Orchestrator의 상태 전이(`DB_SAVED -> GITHUB_QUEUED -> AI_QUEUED -> DONE`)를 관장하고, 멱등성 규칙("DB 먼저, MQ 나중")을 어떤 상황에서도 역순 처리하지 않습니다. Conductor의 에이전트 파일에는 제출 전 검증 순서까지 명시되어 있습니다 — (1) 멤버십 확인, (2) GitHub 연동 확인, (3) 마감 시간 확인, (4) 중복 제출 멱등성 확인.

**Gatekeeper(관문지기)** 는 모든 외부 요청의 유일한 진입점입니다. "당신이 허가하지 않은 요청은 내부 서비스에 도달할 수 없습니다"라고 에이전트 파일에 명시되어 있을 정도로 단호합니다. 보안이 편의보다 항상 우선이고, 의심스러운 요청은 허가하지 않습니다.

**Librarian(기록관리자)** 는 데이터를 지키는 가장 신중한 에이전트입니다. 에이전트 파일의 첫 문장이 인상적입니다 — "데이터는 한 번 잘못 저장되면 복구가 어렵습니다. 당신은 이 프로젝트에서 가장 신중해야 하는 Agent입니다." `down()` 없는 마이그레이션은 작성 자체가 금지되고, Expand-Contract 패턴(3단계 배포)이 강제됩니다.

### Tier 2 — Core

기반 인프라와 핵심 비즈니스 로직을 담당합니다.

**Architect(기반설계자)** 는 k3s 클러스터, CI/CD 파이프라인, 모니터링을 관리합니다. `latest` 태그 이미지 배포 요청은 거부하고, Resource Limit 없는 Pod 배포는 예외 없이 허용하지 않습니다. **Postman(배달부)** 은 GitHub Push를 담당하되, 자신이 "부가 서비스"임을 항상 기억합니다 — "배달이 실패해도 제출 자체는 이미 성공했습니다." **Curator(출제자)** 는 문제 관리와 마감 시간을 통제하고, **Scribe(서기관)** 는 프로젝트의 모든 기억과 문서의 정확성을 전담합니다.

### Tier 3 — Enhancement

서비스의 가치를 높이는 역할입니다. Tier 1, 2가 안정된 후에 작업합니다.

**Sensei(분석가)** 는 Claude API를 호출해서 코드를 분석합니다. Circuit Breaker가 OPEN이면 절대 API를 호출하지 않습니다. **Herald(전령)** 은 프론트엔드 페이지를 구현하되, Palette의 디자인 토큰을 반드시 사용합니다. **Palette(팔레트)** 는 디자인 시스템을 전담하고, `bg-[#...]` 같은 인라인 하드코딩을 금지합니다. **Scout(정찰병)** 는 유일하게 코드를 작성하지 않는 에이전트입니다 — 사용자의 눈으로 서비스를 체험하고 피드백만 보고합니다.

## persona-base.md — 공통 규칙의 SSoT

12개 에이전트가 일관되게 동작하려면 공통 규칙이 필요합니다. AlgoSu에서는 `agents/_shared/persona-base.md`라는 단일 파일이 그 역할을 합니다. SSoT(Single Source of Truth)입니다.

모든 에이전트는 작업 시작 전에 이 파일을 반드시 읽습니다. 에이전트 파일마다 동일한 문구가 있습니다.

> **공통 규칙**: 참조: `agents/_shared/persona-base.md` (착수 전 필수 Read)

이 파일에 정의된 핵심 규칙들을 살펴보겠습니다.

### 보고 체계

<Callout type="info" title="보고 체계 (persona-base.md)">
- Oracle만 PM 직접 소통. Agent의 PM 직접 응답 절대 금지
- 모든 작업은 Oracle 명시적 승인 하에 수행. 독자적 자원 접근 금지
- 작업 완료 후 Oracle 보고. 부여 권한 범위 이탈 금지
</Callout>

에이전트가 "이게 좋을 것 같아서 알아서 했어요"라는 일이 발생하지 않습니다. Oracle의 승인 없이는 아무것도 할 수 없습니다.

### 에스컬레이션 원칙

<Callout type="warn" title="에스컬레이션 원칙">
- 판단 범위를 초과하거나 에이전트 간 충돌 시 → Oracle(심판관)에게 에스컬레이션
- 4시간 내 판단 불가 시 → PM에게 보고
</Callout>

에이전트도 모르는 게 있습니다. 그럴 때 "대충 판단"하지 않고 상위로 올립니다. 4시간 타임아웃은 에이전트 간 의견 충돌이 무한 루프에 빠지는 것을 방지합니다. 실제로 Palette와 Herald 사이에 컴포넌트 스펙 의견 차이가 있을 때, Oracle이 중재하는 구조입니다.

### 인터페이스 계약 원칙

<Callout type="info" title="인터페이스 계약 원칙">
- 변경 24시간 전 관련 에이전트에게 공지
- 하위 호환 우선, 파괴적 변경은 Oracle 승인 필수
- 모든 서비스 간 통신은 X-Internal-Key 헤더 필수
</Callout>

API 스펙을 바꿀 때 사전 공지가 필요하다는 규칙입니다. 마이크로서비스 환경에서 한 서비스의 API 변경이 다른 서비스를 깨뜨리는 것을 방지합니다.

### 코드 작성 규칙

<Callout type="success" title="코드 작성 규칙 (전 에이전트 공통)">
- 클린 코드: 의미 있는 네이밍, 함수 단일 책임, 20줄 이내, DRY
- SOLID 원칙: SRP, OCP, LSP, ISP, DIP
- 주석 필수: 파일 헤더(`@file`, `@domain`, `@layer`, `@related`) + 함수 JSDoc
- 인라인 하드코딩 금지: `bg-[#...]` 사용 금지, Tailwind 토큰 클래스 사용
- JSON structured logging
</Callout>

이 규칙들이 12개 에이전트 전체에 동일하게 적용됩니다. Conductor가 작성한 코드든 Herald가 작성한 코드든 동일한 스타일, 동일한 품질 기준을 따릅니다. `persona-base.md`를 수정하면 모든 에이전트의 행동이 한 번에 바뀝니다.

```text
    ├── conventions.md           # 도구: 코드 컨벤션
    ├── ui.md                    # 도구: UI 디자인 시스템
    └── adr-sprint.md            # 도구: Sprint ADR 템플릿
```

**에이전트 파일**과 **도구 파일**이 분리되어 있습니다. 에이전트 파일(`conductor.md`, `gatekeeper.md` 등)은 역할, 책임, 협업 인터페이스, 판단 기준, 에스컬레이션 조건을 정의합니다. 도구 파일(`annotate.md`, `migrate.md` 등)은 구체적인 규칙과 절차를 담고 있습니다.

각 에이전트는 필요한 도구만 참조합니다. 예를 들어 Librarian의 에이전트 파일에는 이렇게 적혀 있습니다.

<Callout type="info" title="Librarian 도구 참조 (librarian.md 발췌)">
- 마이그레이션 (필독): `agents/commands/migrate.md`
- 어노테이션: `agents/commands/annotate.md`
- 모니터링: `agents/commands/monitor.md`
- CI/CD: `agents/commands/cicd.md`
</Callout>

반면 Scout(정찰병)는 코드를 작성하지 않으므로, 도구 참조가 `ui.md` 하나뿐입니다. 이 분리 덕분에 각 에이전트가 불필요한 규칙에 혼란을 겪지 않습니다.

## 슬래시 커맨드 체계 — PM의 인터페이스

PM(저)이 에이전트 체계와 소통하는 인터페이스는 5개의 슬래시 커맨드입니다. 모두 `.claude/commands/` 디렉토리에 정의되어 있습니다.

### /start — 세션 시작

스프린트를 시작하는 커맨드입니다. 내부적으로 이런 일이 일어납니다.

1. **컨텍스트 로딩**: `sprint-window.md`(슬라이딩 윈도우)를 읽어 이전 스프린트 상태를 파악합니다
2. **상태 검증**: 이전 스프린트가 정상 종료되었는지 확인합니다. `active` 상태면 경고를 띄웁니다
3. **스프린트 제목 결정**: 인자로 전달하거나, 기존 제목을 유지하거나, 사용자에게 질문합니다
4. **스프린트 활성화**: `sprint-window.md`의 상태를 `active`로 변경하고, 현재 HEAD commit을 기록합니다
5. **대시보드 출력**: 현재/이전 스프린트 상태, 잔여 항목, 작업 안내를 보여줍니다

대시보드 출력의 마지막에는 항상 이 안내가 나옵니다.

> 모든 작업은 `/algosu-oracle`을 통해 요청하세요.
> Oracle이 12개 전문 에이전트에게 적절히 위임합니다.

### /stop — 세션 종료

스프린트를 종료하는 커맨드입니다. 6단계로 구성되어 있고, 각 단계가 실패해도 복구 가능하도록 설계되어 있습니다.

1. **현재 상태 확인**: `sprint-window.md` 읽기 + `git log` 실행. 커밋이 없으면 취소할지 물어봅니다
2. **자동 초안 + 사용자 확인**: git log 기반으로 작업 요약을 자동 생성하고, PM에게 수정 기회를 줍니다
3. **Sprint ADR 생성**: `docs/adr/sprints/sprint-{N}.md` 파일을 생성합니다. 결정, 패턴, 교훈을 기록합니다
4. **슬라이딩 윈도우 갱신**: 완료 스프린트를 `[1] 완료`로 승격하고, 새 `[2] 다음` 섹션을 작성합니다
5. **MEMORY.md 갱신 + 일관성 검증**: 테이블 갱신, 3개 파일 일관성 점검, 모든 검증 후에만 `idle`로 전환
6. **완료 보고**: 종료 결과를 요약해서 PM에게 보고합니다

핵심 설계 원칙은 "상태 전환은 반드시 모든 갱신이 완료된 후 마지막에 수행한다"입니다. `status: idle`로의 전환을 마지막에 하기 때문에, 중간에 중단되어도 `/stop`을 다시 실행하면 누락된 단계부터 재개할 수 있습니다. Saga 패턴과 비슷한 발상입니다.

### /algosu-oracle — 작업 요청

Oracle에게 작업을 요청하는 핵심 커맨드입니다. Oracle은 요청을 받으면 이렇게 동작합니다.

1. `agents/_shared/persona-base.md`를 읽어 공통 규칙을 확인합니다
2. 작업 내용을 분석하고, 필요한 에이전트를 식별합니다
3. 해당 에이전트의 파일(`agents/commands/{name}.md`)을 읽어 역할을 이해합니다
4. 에이전트에게 작업을 할당하고, 결과를 취합합니다
5. PM에게 결과를 보고합니다

Oracle의 파일에 에이전트 참조 경로가 명시되어 있습니다.

<Callout type="info" title="Oracle 에이전트 참조 경로 (oracle.md 발췌)">
작업 할당 시 해당 에이전트 파일을 Read하여 역할을 수행하세요:
`agents/commands/{name}.md` — conductor, gatekeeper, librarian, architect, scribe, postman, curator, herald, palette, scout, sensei
</Callout>

### /algosu-review — 코드 리뷰

코드 리뷰 전용 커맨드입니다. 공통 체크리스트, 보안 체크리스트, 마이그레이션 체크리스트를 통해 변경 사항을 검증합니다. PR 유형에 따라 승인 기준도 다릅니다.

| 유형 | 승인 | 필수 |
|------|------|------|
| 일반 | 1명 | CODEOWNERS |
| 보안 | 2명 | Gatekeeper |
| 인프라 | 1명 | Architect |
| DB 스키마 | 2명 | Architect + Conductor |
| Breaking | 2명 | Oracle 최종 |

보안 관련 변경은 반드시 Gatekeeper의 검토가 필요하고, DB 스키마 변경은 Architect와 Conductor 둘 다의 승인이 필요합니다. Oracle이 최종 결정권을 갖는 건 Breaking Change뿐입니다.

### /algosu-adr-blog — ADR을 블로그로 변환

정식 ADR이나 Sprint ADR을 기술 블로그 MDX 포스트로 변환하는 커맨드입니다. 기술 보고서 톤의 ADR을 "읽기 쉬운 블로그 글"로 변환하되, 코드 블록과 다이어그램은 유지합니다. 이 글도 그 체계의 산출물입니다.

## 실제 스프린트에서의 오케스트레이션

구체적인 예시로 Sprint 63~64를 보겠습니다. Discord 피드백 알림 기능을 추가하고 버그를 수정한 스프린트입니다.

**Sprint 63 — Discord 피드백 알림 (2커밋, 10파일)**

```text
PM: "관리자가 피드백을 남기면 Discord로 알림이 가게 해줘"

Oracle 작업 분석:
  - 이건 새 기능 추가다
  - Identity 서비스에 Discord webhook 로직 필요 (Conductor 영역? 아니, Identity)
  - Gateway에 알림 트리거 추가 필요 (Gatekeeper 영역)
  - 테스트 추가 필요

Oracle 할당:
  1. Gatekeeper → 피드백 생성 시 webhook 트리거 API
  2. Conductor → Identity 서비스에 Discord webhook 로직 구현
  3. Scribe → Sprint ADR 기록
```

**Sprint 64 — Discord 알림 미전송 버그 수정 (3커밋, 8파일)**

바로 다음 스프린트에서 "알림이 안 간다"는 버그가 발견됐습니다. 이때 Oracle의 우선순위 원칙이 작동합니다 — 서비스 안정성이 최우선. 새 기능 개발을 중단하고 버그 수정부터 진행합니다. 추가로 "피드백 해결 시 알림"이라는 누락된 케이스도 함께 처리했습니다.

이렇게 스프린트 단위로 작업이 진행되고, 매 스프린트마다 `/stop` 커맨드로 Sprint ADR이 생성되며, `MEMORY.md`에 이력이 축적됩니다. 67개 스프린트가 이 사이클을 반복했습니다.

## 에이전트 모델 배치 — 역할에 맞는 모델

흥미로운 점은 모든 에이전트가 같은 AI 모델을 사용하지 않는다는 것입니다. 각 에이전트 파일의 frontmatter에 모델이 지정되어 있습니다.

| 모델 | 에이전트 | 이유 |
|------|---------|------|
| `claude-opus-4-6` | Oracle, Conductor, Gatekeeper, Librarian | 복잡한 판단, 보안 결정, 데이터 정합성 |
| `claude-sonnet-4-6` | Architect, Postman, Curator, Scribe, Sensei, Herald, Palette, Scout | 구현 작업, 비용 효율 |

Tier 1의 Mission Critical 에이전트들은 더 강력한 모델(Opus)을 사용합니다. 기획 결정, 보안 판단, 데이터 스키마 설계처럼 실수하면 큰 비용이 드는 작업이기 때문입니다. Tier 2~3 에이전트들은 Sonnet으로 충분합니다. 구현 작업은 규칙만 잘 따르면 되니까요.

## 이 구조의 장점

**1. 일관된 코드 품질**

67개 스프린트를 거치면서 코드 스타일이 흔들리지 않았습니다. `persona-base.md`가 SSoT로 작동하기 때문입니다. 어노테이션 규칙, 네이밍 컨벤션, 에러 핸들링 패턴이 처음부터 끝까지 동일합니다.

**2. 맥락 유실 방지**

Sprint ADR + `MEMORY.md` + `sprint-window.md` 3중 구조가 맥락을 보존합니다. 3개월 전 결정을 5초 만에 찾을 수 있습니다. Scribe가 200줄 이내로 메모리를 관리하기 때문에 항상 최신 맥락만 빠르게 파악됩니다.

**3. 안전한 변경**

Librarian이 DB 변경을 3단계 Expand-Contract로 강제하고, Gatekeeper가 보안 규칙을 검증하고, Architect가 리소스 제한을 확인합니다. 한 에이전트의 변경이 다른 영역을 깨뜨리는 것을 구조적으로 방지합니다.

**4. 확장 가능한 구조**

새로운 역할이 필요하면 `agents/commands/` 에 에이전트 파일을 하나 추가하면 됩니다. 공통 규칙은 `persona-base.md`를 읽으면 자동으로 적용됩니다.

## 이 구조의 한계

솔직하게 한계도 이야기해야 합니다.

**1. 컨텍스트 윈도우 비용**

에이전트가 작업을 시작할 때마다 `persona-base.md`를 읽고, 자기 에이전트 파일을 읽고, 도구 파일을 읽습니다. 이 "사전 로딩"이 컨텍스트 윈도우를 소비합니다. 복잡한 작업에서는 실제 코드를 읽을 여유가 줄어들 수 있습니다.

**2. Oracle 병목**

모든 작업이 Oracle을 거치므로, Oracle이 병목이 될 수 있습니다. 동시에 여러 에이전트에게 작업을 할당하는 것은 가능하지만, 최종 검증은 Oracle이 순차적으로 처리합니다.

**3. 규칙 유지 비용**

에이전트 파일, 도구 파일, 공통 규칙을 최신 상태로 유지하는 것도 비용입니다. 프로젝트가 진화하면 규칙도 함께 진화해야 합니다. Scribe가 이 역할을 담당하지만, 최종 판단은 여전히 PM(저)의 몫입니다.

**4. 도메인 직관의 부재**

"이 기능이 사용자에게 정말 필요한가?"는 에이전트가 답할 수 없습니다. 기술 결정의 집행은 에이전트에게 맡길 수 있지만, 방향 결정은 사람이 해야 합니다.

## 마무리 — 통제된 AI가 유용한 AI다

AI에게 코드를 맡기는 것 자체는 어렵지 않습니다. 어려운 건 그 AI가 프로젝트의 규칙을 지키고, 다른 영역과 충돌하지 않고, 실수했을 때 되돌릴 수 있도록 만드는 것입니다.

AlgoSu의 오케스트레이션 구조는 이 문제에 대한 하나의 답입니다. 완벽한 답은 아니지만, 67개 스프린트를 일관된 품질로 완주할 수 있게 해준 구조입니다. Oracle이 판단하고, 11개 에이전트가 실행하고, `persona-base.md`가 일관성을 유지합니다. 통제 없는 AI는 위험하지만, 통제된 AI는 팀원보다 규칙을 잘 지킵니다.
