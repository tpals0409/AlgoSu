## 혼자서 코드리뷰를 받을 수 있을까요?

팀 프로젝트에서 가장 먼저 사라지는 건 코드리뷰입니다. 1인 개발이라면 처음부터 없죠. 새벽 2시에 커밋한 코드를 다음 날 아침에 다시 보면 "이걸 왜 이렇게 짰지?" 하는 순간이 찾아옵니다. 맥락은 증발하고, 품질은 운에 맡기게 됩니다.

저는 알고리즘 스터디 관리 플랫폼 AlgoSu를 혼자 만들고 있었습니다. 6개 마이크로서비스, 3개 데이터베이스, RabbitMQ, Redis, Kubernetes 배포까지. 기능이 늘어날수록 한 사람의 머릿속에 전체 시스템을 담아두기가 불가능해졌습니다.

그래서 AI 에이전트 12명을 "팀원"으로 영입했습니다.

## 1인 개발의 세 가지 벽

### 1. 코드리뷰가 없다

PR을 올려도 봐줄 사람이 없습니다. "LGTM"을 스스로에게 날리는 건 리뷰가 아닙니다. JWT 검증 로직을 수정하면서 Rate Limit 가드를 빼먹어도 아무도 지적하지 않습니다.

### 2. 맥락이 유실된다

화요일에 Saga Orchestrator 패턴을 설계하고, 수요일에 프론트엔드 SSE 연동을 하고, 목요일에 k8s 매니페스트를 수정합니다. 금요일에 다시 Saga로 돌아오면 화요일의 결정 이유가 기억나지 않습니다.

### 3. 품질 기준이 흔들린다

"이번만 하드코딩하자", "테스트는 나중에 작성하자". 혼자면 이런 유혹을 막아줄 사람이 없습니다. 기술 부채는 눈덩이처럼 불어납니다.

저(PM)는 Oracle하고만 대화합니다. Oracle이 작업을 분석해서 적절한 에이전트에게 위임합니다. 에이전트들은 서로 직접 소통하지 않고, 충돌이 생기면 Oracle이 중재합니다.

## 12명의 에이전트, 각자의 전문 영역

에이전트 체계를 설계할 때 가장 신경 쓴 부분은 **책임의 경계**입니다. 실제 개발 팀에서도 백엔드 개발자가 디자인 시스템을 건드리면 문제가 생기듯, 에이전트도 자기 영역 밖의 일을 하면 안 됩니다.

| 에이전트 | 역할 | 실제 팀 비유 | Tier |
|---------|------|------------|------|
| **Oracle** | 기획 결정, 에이전트 조율, ADR 관리 | PM + Tech Lead | 1 |
| **Conductor** | 제출 생명주기, Saga 오케스트레이션 | 백엔드 리드 | 1 |
| **Gatekeeper** | 인증, JWT, OAuth, 보안 | 보안 엔지니어 | 1 |
| **Librarian** | DB 스키마, 마이그레이션, 데이터 정합성 | DBA | 1 |
| **Architect** | 인프라, k8s, CI/CD, 모니터링 | DevOps | 2 |
| **Postman** | GitHub 연동, MQ 소비 | 백엔드 개발자 | 2 |
| **Curator** | 문제 관리, 마감 시간, 통계 | 백엔드 개발자 | 2 |
| **Scribe** | 문서, 메모리, 프롬프트 관리 | 테크니컬 라이터 | 2 |
| **Sensei** | AI 분석, Claude API, Circuit Breaker | ML 엔지니어 | 3 |
| **Herald** | 프론트엔드 페이지, SSE, UX | 프론트엔드 개발자 | 3 |
| **Palette** | 디자인 시스템, 컴포넌트, 접근성 | UI 디자이너 | 3 |
| **Scout** | 사용자 관점 테스트, UX 피드백 | QA 엔지니어 | 3 |

Tier는 우선순위입니다. 서비스 안정성(Tier 1) > 개발 속도(Tier 2) > 기능 완성도(Tier 3). 리소스가 부족할 때 뭘 먼저 할지 자동으로 결정됩니다.

## 공통 규칙: persona-base.md

12명이 제각각 움직이면 안 됩니다. `agents/_shared/persona-base.md`라는 공통 규칙 파일이 모든 에이전트의 행동 기준을 잡아줍니다. 핵심 규칙 몇 가지를 소개합니다.

**보고 체계가 명확합니다.**

> Oracle만 PM 직접 소통. Agent의 PM 직접 응답 절대 금지.
> 모든 작업은 Oracle 명시적 승인 하에 수행. 독자적 자원 접근 금지.

에이전트가 독자적으로 판단해서 코드를 바꾸는 일이 없습니다. 반드시 Oracle을 거칩니다.

**에스컬레이션 원칙도 정의되어 있습니다.**

> 판단 범위를 초과하거나 에이전트 간 충돌 시 Oracle에게 에스컬레이션.

Palette(디자인)와 Herald(프론트엔드)가 컴포넌트 스펙에서 의견이 다르면? Oracle이 중재합니다. 4시간 이상 미결이면 PM(저)한테 올라옵니다.

**코드 작성 규칙은 모든 에이전트에게 동일합니다.**

함수 단일 책임, 20줄 이내, DRY, SOLID 원칙. 파일 헤더에 `@file`, `@domain`, `@layer`, `@related` 어노테이션 필수. 인라인 하드코딩(`bg-[#...]`) 금지. 이런 규칙들이 일관된 코드 품질을 유지하게 해줍니다.

## 에이전트 도입 전/후 비교

솔직하게 비교해보겠습니다.

### 전: "혼자서 다 하기"

- 커밋 메시지: `fix stuff`, `wip`, `asdf`
- 코드리뷰: 없음
- 테스트: "나중에"
- 문서: 머릿속에만 존재
- 맥락 유실: 매번 코드를 처음부터 다시 읽음
- DB 마이그레이션: `synchronize: true`로 퉁치기
- 보안: "일단 동작하면 됨"

### 후: "12명과 함께하기"

- 커밋 메시지: `feat|fix|chore(<scope>): <subject>` 컨벤션 강제
- 코드리뷰: Gatekeeper가 보안 체크, Scout가 UX 피드백
- 테스트: **2,432개** (전 서비스 커버리지 목표 달성)
- 문서: Scribe가 Sprint ADR + 메모리 자동 관리
- 맥락 유실: `MEMORY.md` + Sprint ADR에 모든 결정 기록
- DB 마이그레이션: Librarian이 Expand-Contract 패턴 강제, `down()` 없는 마이그레이션 작성 금지
- 보안: Gatekeeper가 JWT none 알고리즘 금지, 민감 정보 마스킹 필수 규칙 적용

## 67개 스프린트의 결과물

Sprint 3부터 67까지, 65개 스프린트를 이 체계로 진행했습니다. 숫자로 정리하면 이렇습니다.

| 지표 | 수치 |
|------|------|
| 총 스프린트 | 65회 (Sprint 3~67) |
| 마이크로서비스 | 6개 (NestJS 5 + FastAPI 1) |
| 프론트엔드 | Next.js 15 (App Router) |
| 테스트 | 2,432건 |
| CI Jobs | 15개 (matrix) |
| 데이터베이스 | 3개 (Database per Service) |
| 배포 | GitOps (ArgoCD → k3s, OCI ARM) |
| 프로젝트 완성도 | ~99% |

몇 가지 이정표를 꼽아보면:

- **Sprint 3~10**: 기본 CRUD, OAuth 로그인, DB 분리 (ADR-001)
- **Sprint 20대**: Saga Orchestrator, RabbitMQ 도입, GitHub 연동
- **Sprint 40대**: AI 분석 Circuit Breaker, SSE 실시간 알림
- **Sprint 50대**: 관리자 페이지, 피드백 시스템, Discord 알림 연동
- **Sprint 60대**: 보안 강화, 인앱 알림 고도화, 기술 블로그

혼자서 이 규모의 프로젝트를 일관된 품질로 유지할 수 있었던 건, 에이전트 체계가 "다음에 하자"를 "지금 해야 한다"로 바꿔줬기 때문입니다.

## 실제로 효과가 있었던 순간들

### Librarian이 막아준 DB 사고

Problem 서비스의 스키마를 변경할 때였습니다. 컬럼 이름을 바꾸려고 했는데, Librarian의 규칙이 이를 막았습니다.

> Expand-Contract 패턴 강제. 컬럼 삭제/rename은 반드시 3단계 배포.
> Rolling Update 중 구/신 버전 공존 상황을 항상 가정.

단순 rename 대신 (1) 새 컬럼 추가 → (2) 데이터 복사 + 앱 코드 전환 → (3) 구 컬럼 삭제의 3단계로 진행했습니다. 운영 중 다운타임 없이 완료할 수 있었습니다.

### Gatekeeper의 보안 원칙

인증 로직을 리팩터링하면서 localStorage에 토큰을 저장하는 코드가 남아 있었습니다. Gatekeeper의 규칙이 이를 잡아냈습니다.

> SSE 인증: `EventSource(url, { withCredentials: true })` — httpOnly Cookie 환경에서 localStorage 토큰 사용 불가.
> 민감 정보 절대 금지: JWT 원문, X-Internal-Key, OAuth 토큰.

Sprint 65에서 localStorage 토큰 잔재를 전부 정리했습니다.

### Scribe의 기억력

67개 스프린트를 거치면서 "이전에 왜 이렇게 결정했지?" 하는 순간이 수도 없이 있었습니다. Scribe가 관리하는 `MEMORY.md`와 Sprint ADR에 모든 결정 기록이 남아 있었기 때문에, 3개월 전 결정도 5초 만에 찾을 수 있었습니다.

> MEMORY.md 200줄 이내 유지. 완료 Sprint는 토픽 파일 이동.
> 2 Sprint 이전 상세는 삭제 또는 archive 이동.

메모리를 계속 정리하기 때문에 항상 최신 맥락만 빠르게 파악할 수 있습니다.

## 이 체계를 만들면서 배운 것들

### 1. 역할 경계가 생산성이다

에이전트에게 "다 할 수 있어"라고 하면 품질이 떨어집니다. "너는 DB만 봐"라고 하면 그 영역에서 깊이 있는 판단을 합니다. Palette에게 "API도 수정해줘"라고 하면 안 됩니다. Palette는 디자인 시스템만 알면 됩니다.

### 2. 공통 규칙은 SSoT로 관리해야 한다

`persona-base.md` 하나에 공통 규칙을 모아두고, 모든 에이전트가 작업 시작 전에 이 파일을 읽습니다. 규칙이 에이전트별로 흩어지면 불일치가 생기고, 예외가 쌓이고, 결국 규칙이 무력화됩니다.

### 3. 에스컬레이션 경로가 있어야 한다

에이전트도 판단이 안 되는 상황이 있습니다. "이 API 스펙을 바꿔야 할까?"같은 기획 결정은 Oracle로, Oracle도 판단이 안 되면 PM(저)에게 올라옵니다. 이 경로가 없으면 에이전트가 독자적으로 잘못된 결정을 내립니다.

### 4. Tier 분류가 리소스 충돌을 예방한다

모든 에이전트가 동시에 같은 파일을 수정하면 충돌이 납니다. Tier 1(Conductor, Gatekeeper, Librarian)이 먼저, 그 다음 Tier 2, Tier 3 순서로 작업하면 의존성 충돌이 줄어듭니다.

## 마무리: 혼자이지만 혼자가 아닌 개발

1인 개발의 가장 큰 약점은 "아무도 지적해주지 않는다"는 것입니다. AI 에이전트 체계는 이 문제를 구조적으로 해결합니다. 에이전트는 지치지 않고, 규칙을 잊지 않고, 감정적으로 "이번만 넘어가자"라고 하지 않습니다.

물론 한계도 있습니다. 에이전트는 비즈니스 도메인에 대한 직관이 없습니다. "이 기능이 사용자에게 정말 필요한가?"는 여전히 사람이 판단해야 합니다. 기술 결정의 집행은 에이전트에게, 방향 결정은 사람에게. 이 분업이 67개 스프린트를 버틸 수 있게 해준 비결입니다.

다음 글에서는 이 에이전트들이 실제로 관리하는 [AlgoSu의 전체 시스템 아키텍처](/posts/system-architecture-overview)를 살펴봅니다. 6개 마이크로서비스가 하나의 제출을 어떻게 처리하는지, 조감도 시점에서 그려보겠습니다.
