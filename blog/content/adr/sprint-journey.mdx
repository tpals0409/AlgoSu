## 73번의 스프린트, 약 2,453개의 테스트

숫자부터 꺼내겠습니다.

- **73개** 스프린트 (Sprint 3 ~ 73, 총 71회)
- **약 2,453개** 테스트 케이스
- **6개** 마이크로서비스 + **1개** 프론트엔드
- **3개** 데이터베이스 (identity_db, problem_db, submission_db)
- **15개** CI job
- **12개** AI 에이전트
- **34개** Identity API 엔드포인트 (Sprint 51에서 0개 → 34개)
- **Identity branches 커버리지** 99.51%

Sprint 3에서 시작한 AlgoSu가 Sprint 73에서 "약 99% 완성"이라는 말을 할 수 있게 되기까지, 저는 네 번의 큰 전환점을 겪었습니다. 이 글은 그 여정의 기록입니다.

> 이 글은 [AlgoSu 기술 블로그 시리즈](/posts/agent-orchestration-solo-dev) 6편 중 마지막(6편)입니다.

처음에는 당연히 "일단 돌아가게 만들자"였습니다. Problem CRUD, Submission CRUD, 사용자 인증. 가장 기본적인 것들이요.

### OAuth 삽질기

OAuth는 세 개를 한꺼번에 붙였습니다. Google, Naver, Kakao. 각 플랫폼의 콜백 URL 형식이 다르고, 반환하는 프로필 데이터 구조도 다르고, 토큰 갱신 정책도 달랐습니다. "OAuth 붙이는 건 쉽다"는 말은 하나만 붙일 때의 이야기예요.

하지만 이때 내린 결정 하나가 나중에 큰 도움이 됐습니다 — **httpOnly Cookie JWT**. localStorage에 토큰을 저장하는 대신 httpOnly 쿠키를 선택한 건 XSS 공격 표면을 줄이기 위해서였는데, Sprint 65에서 localStorage 토큰 함수 11개를 대량 제거할 때 "초기에 잘 결정했다"는 생각이 들었습니다.

### Database per Service

Sprint 12 즈음에 3개 데이터베이스를 분리했습니다. identity_db, problem_db, submission_db. 솔직히 이 시점에서는 과하다고 느꼈습니다. 서비스가 3개밖에 없는데 DB를 왜 3개로 나누나 싶었죠.

하지만 Sprint 51에서 Gateway → Identity DB 분리를 할 때, 이미 DB가 나뉘어 있어서 마이그레이션이 훨씬 수월했습니다. 초기의 "과한" 설계가 결국 미래의 저를 구해준 셈입니다.

### Phase 1의 교훈
> **"나중에 할 일을 줄이려면 지금 조금 더 해야 한다."**
> DB 분리, httpOnly Cookie, 외부 ID(publicId) 도입 — 초기에 조금 더 시간을 들인 결정들이 이후 수십 개의 스프린트에서 누적으로 시간을 아껴줬습니다.

<PhaseTimeline>
  <PhaseMilestone phase="Sprint 16" title="Saga Orchestrator 패턴 도입" period="Submission Service 상태 머신" status="done" icon="Workflow" accent={1} />
  <PhaseMilestone phase="Sprint 22" title="RabbitMQ 이벤트 버스" period="비동기 워크플로우 전환" status="done" icon="Network" accent={2} />

  <PhaseMilestone phase="Sprint 35" title="AI 분석 + Circuit Breaker" period="Claude API · FastAPI" status="done" icon="Sparkles" accent={4} />
</PhaseTimeline>

Phase 2는 "모놀리스 같은 마이크로서비스"에서 "진짜 마이크로서비스"로 가는 과정이었습니다. 서비스들이 물리적으로는 나뉘어 있었지만, 실질적으로는 동기 HTTP 호출로 강하게 결합되어 있었거든요.

### Saga Orchestrator

코드 제출 워크플로우가 복잡해졌습니다. 사용자가 코드를 제출하면:

1. Submission 서비스가 제출을 저장하고
2. GitHub Worker가 GitHub에 Push하고
3. AI Analysis가 코드를 분석하고
4. 결과가 다시 Submission으로 돌아와야 합니다

이 과정에서 하나라도 실패하면? 부분적으로 완료된 상태가 남으면 안 됩니다. Saga Orchestrator 패턴을 도입해서 Submission 서비스가 전체 워크플로우의 상태를 관리하도록 했습니다.

### RabbitMQ 도입

Saga를 구현하다 보니 서비스 간 비동기 통신이 필요해졌습니다. RabbitMQ를 도입하고 이벤트 기반 아키텍처로 전환했습니다. Submission → RabbitMQ → GitHub-Worker(Push) / AI-Analysis(분석) 흐름이 이때 만들어졌어요.

### AI 분석과 Circuit Breaker

AI Analysis 서비스는 Claude API를 호출합니다. 외부 API는 언제든 느려지거나 실패할 수 있죠. Circuit Breaker를 넣어서 Claude API가 연속으로 실패하면 일정 시간 동안 요청을 차단하도록 했습니다. 이건 FastAPI + Python으로 별도 서비스로 만들었는데, NestJS 통일성을 깨는 대신 Python ML 생태계의 이점을 가져간 트레이드오프였습니다.

### Phase 2의 교훈
> **"비동기는 디버깅이 10배 어렵다."**
> 동기 호출에서는 에러가 즉시 터집니다. 비동기 이벤트 기반에서는 "메시지가 발행됐는데 왜 소비가 안 되지?"를 추적하는 게 정말 고됐습니다. 하지만 비동기 없이는 서비스 간 결합도를 끊을 수 없었습니다.

</PhaseTimeline>

Phase 3는 "돌아가는 코드"에서 "운영 가능한 시스템"으로 가는 과정이었습니다. 그리고 이 Phase에서 가장 큰 사건이 일어났습니다.

### CI 15 jobs의 탄생

처음에는 CI가 3개 job이었습니다. lint, test, build. AI가 코드를 빠르게 생성하니까 CI도 간단하면 될 줄 알았어요.

**틀렸습니다.**

`git commit --amend` 후 force-push를 했더니, `dorny/paths-filter`가 변경을 감지하지 못해서 빌드가 통째로 건너뛰어진 사건이 있었습니다. 이후 force-push 금지 규칙과 함께 CI를 점진적으로 강화했고, 최종적으로 15개 job까지 늘어났습니다.

secret-scan, detect-changes, quality checks(NestJS/Python/Frontend), 테스트(Jest/pytest/Vitest), audit, build, GitOps sync까지. AI가 빠르게 코드를 만들어줄수록, 검증 파이프라인은 더 촘촘해야 한다는 교훈을 얻었습니다.

### ADR-001: Gateway → Identity DB 분리 (Sprint 51)

이건 별도의 ADR 문서를 작성할 만큼 중요한 결정이었습니다. Gateway 서비스가 identity_db에 직접 접근하는 구조는 DB per Service 원칙을 위반하고 있었거든요. 6개 엔티티(User, Study, StudyMember, StudyInvite, Notification, ShareLink)를 19개 파일에서 직접 CRUD하고 있었습니다.

결정은 전면 이관이었습니다:
- Identity 서비스에 **34개 API 엔드포인트**를 신규 구축
- Gateway의 19개 파일을 모두 **IdentityClient HTTP 호출**로 전환
- Entity 파일 4개 삭제, 공통 타입으로 분리

k8s 내부 네트워크 호출이므로 레이턴시 추가는 약 1ms. Gateway branches 커버리지는 97.79%를 유지했습니다. 이게 Sprint 51 하나에서 일어난 일입니다.

### GitOps 자동화

ArgoCD와 k3s를 연결해서 완전한 GitOps 파이프라인을 구축했습니다. `git push` → CI 15 jobs → ghcr.io 이미지 빌드 → aether-gitops 레포 태그 갱신 → ArgoCD → k3s 배포. 사람이 수동으로 개입하는 단계가 사라졌습니다.

### Phase 3의 교훈
> **"자동화에 투자한 시간은 반드시 돌아온다."**
> CI 15 jobs를 만드는 데 여러 스프린트가 걸렸지만, 이후 모든 스프린트에서 "이 커밋이 안전한가?"를 사람이 판단할 필요가 없어졌습니다. 그리고 ADR-001처럼 큰 리팩토링도 테스트 커버리지가 받쳐주니까 과감하게 실행할 수 있었습니다.

### Sprint 62: 관리자 페이지의 교훈

피드백 관리 페이지에서 status 필터를 클라이언트에서 처리하고 있었습니다. 서버 페이지네이션(20건/페이지)과 조합하면 어떻게 될까요? 현재 페이지의 20건 안에서만 필터링되고, 다른 페이지의 해당 상태 피드백은 보이지 않습니다.

당연한 버그인데, 만들 때는 인지하지 못했어요. 서버 사이드 필터로 전환하고, 응답에 전체 기준 counts 통계를 포함시켜서 해결했습니다. 8커밋, 17파일이 변경됐습니다.

스크린샷 저장도 JPEG에서 WebP로 바꿨습니다. Base64 Data URL로 DB TEXT 컬럼에 저장하는 구조라 용량이 중요했거든요. `canvas.toDataURL('image/webp', 0.65)`로 JPEG 대비 25~35% 절감을 얻었습니다.

### Sprint 63~64: Discord 알림의 함정

Discord 피드백 알림은 기술적으로 단순했습니다. Identity 서비스에 DiscordWebhookService를 만들고, 피드백 생성 시 webhook을 쏘면 끝. fire-and-forget 패턴으로 메인 로직에 영향 없이요.

**함정은 배포에 있었습니다.**

Sprint 63에서 코드를 완성하고 push했습니다. Secret도 만들었고, 소스 레포 매니페스트도 작성했습니다. 그런데 Discord 알림이 오지 않았어요. 원인은? **aether-gitops 레포에 env 블록을 반영하지 않았던 겁니다.** ArgoCD가 동기화하는 건 GitOps 레포의 매니페스트인데, 소스 레포만 수정하고 GitOps 레포를 잊어버린 거죠.

Sprint 64에서 이걸 수정하면서 추가로 `FEEDBACK_RESOLVED` 인앱 알림도 구현했는데, 여기서 또 한번 빠졌습니다. Identity 엔티티에 enum을 추가했지만 Gateway enum과 Frontend type에 전파하지 않은 겁니다. 결국 Sprint 66에서 3-tier(Identity → Gateway → Frontend) 전체를 동기화해야 했습니다.

### Sprint 65: 보안 점검

httpOnly Cookie로 전환한 지 한참 됐는데도 localStorage 관련 코드가 남아 있었습니다. `setToken`, `getToken`, `isTokenExpired` 등 **11개 함수**를 제거했습니다. 프로덕션에서는 이미 미사용이었지만, 테스트 코드에서 참조하고 있어서 같이 정리해야 했어요.

`DEV_MOCK` 바이패스에 `NODE_ENV` 가드를 추가한 것도 이 스프린트입니다. `NEXT_PUBLIC_DEV_MOCK=true`가 프로덕션 빌드에 번들링되면 Edge Middleware 전체가 우회되는 잠재적 위험이 있었거든요.

1커밋으로 7개 파일을 바꿨는데, **+42줄 / -440줄**이었습니다. 코드를 삭제하는 스프린트가 이렇게 상쾌할 줄은 몰랐습니다.

### Sprint 67: 기술 블로그

스프린트 페이지를 제거하고 ADR 기반 기술 포스트로 대체했습니다. 그리고 지금 이 글을 쓰고 있습니다. blog.yaml GitOps 매니페스트의 imagePullPolicy와 GHCR 설정도 함께 동기화했어요.

### Phase 4의 교훈
> **"enum 하나 추가하면 5개 파일이 바뀐다."**
> MSA에서 타입 동기화는 생각보다 손이 많이 갑니다. Sprint 66에서 NotificationType 추가 시 전파 체크리스트(Identity entity → Gateway types → Frontend api.ts → NotificationBell → NotificationToast)를 만들었는데, 처음부터 있었으면 Sprint 64의 삽질은 없었을 겁니다.

## 전환점이 된 스프린트들

### 1. Sprint 12: Database per Service 확정

"지금은 과한 것 같지만 나중에 감사할 거야"라고 스스로를 설득했던 순간. 실제로 Sprint 51의 ADR-001 작업이 이 결정 덕에 한결 수월했습니다. 미래의 나에게 선물을 준 셈이었어요.

### 2. Sprint 22: RabbitMQ 이벤트 버스 도입

동기 HTTP 호출의 한계를 체감한 스프린트. Submission → GitHub Push → AI 분석의 비동기 파이프라인이 이때 탄생했습니다. 디버깅은 어려워졌지만, 서비스 간 결합도는 극적으로 낮아졌습니다.

### 3. Sprint 51: ADR-001 Gateway → Identity DB 분리

프로젝트에서 가장 큰 단일 리팩토링. 0개였던 Identity API가 34개로 늘어나고, Gateway의 19개 파일이 HTTP 클라이언트로 전환됐습니다. 테스트 커버리지가 받쳐줬기 때문에 가능한 작업이었어요.

### 4. Sprint 63~64: GitOps 매니페스트 동기화의 교훈

코드는 완벽하게 작성했는데 배포가 안 됐던 사건. "소스 레포 매니페스트를 수정하면 GitOps 레포도 반드시 함께 갱신한다"는 규칙이 이때 확립됐습니다. 이후 `kubectl exec -- printenv`로 실제 주입 검증하는 습관도 이 스프린트에서 생겼어요.

### 5. Sprint 65: 코드 삭제의 쾌감

+42줄 / -440줄. localStorage 토큰 잔재를 정리하고, DEV_MOCK 보안 가드를 추가한 스프린트. 기능을 추가하는 것보다 불필요한 코드를 제거하는 게 시스템을 더 안전하게 만든다는 걸 체감했습니다.

## 잘한 것 vs 아쉬운 것

### 잘한 것

**1. 초기부터 DB를 분리한 것.** Phase 3에서 큰 리팩토링을 할 때 DB가 이미 분리되어 있어서 작업 범위가 명확했습니다.

**2. 테스트에 집착한 것.** 약 2,453개의 테스트와 99.51%의 branches 커버리지. 이게 없었으면 ADR-001 같은 대규모 리팩토링은 엄두도 못 냈을 겁니다.

**3. ADR을 작성한 것.** 중요한 결정마다 왜 그 선택을 했는지 기록해뒀더니, 나중에 비슷한 상황에서 같은 고민을 반복하지 않았습니다. Sprint ADR은 프로젝트의 영구 기록이 됐습니다.

**4. AI 에이전트 체계를 만든 것.** 12개 에이전트가 각자의 영역을 책임지니까, 저는 "무엇을 해야 하는지"에만 집중할 수 있었습니다. "어떻게 해야 하는지"는 에이전트가 알아서 처리했어요.

### 아쉬운 것

**1. TLS를 아직 도입하지 못한 것.** 73개 스프린트를 거치면서도 TLS 인증서 설정은 계속 미뤄졌습니다. 기능 구현에 우선순위를 뒀기 때문인데, 보안 관점에서는 더 일찍 했어야 합니다.

**2. MSA 타입 동기화 자동화 부재.** enum 하나 추가할 때마다 5개 파일을 수동으로 바꿔야 합니다. 코드 생성기나 공유 패키지를 도입했으면 Sprint 64/66 같은 동기화 누락 사고는 없었을 거예요.

**3. 모니터링 알림을 늦게 붙인 것.** alertmanager를 설정해놨으면서 실제 알림 채널(Discord)을 Sprint 63까지 연결하지 않았습니다. "모니터링 도구가 있다"와 "모니터링이 동작한다"는 다른 문제입니다.

## 다음은 뭘 할까?

Sprint 73까지 오면서 AlgoSu는 "약 99% 완성"이라고 말할 수 있는 수준이 됐습니다. 하지만 아직 남은 것들이 있습니다.

**비차단 후속 작업들:**
- TLS 인증서 도입
- 기존 문제 DB level=2 데이터 수정 (AddProblemModal 버그 잔여)
- 기존 사용자 GitHub 토큰 재연동

**아키텍처 검토 대상:**
- 타 서비스(submission, problem, github-worker) → Identity 직접 호출 전환 (현재 Gateway 프록시)
- postgres 인스턴스 물리 분리 검토 (OCI ARM 리소스 한계)
- React Query 도입 여부 (현재 순수 Context + hooks)
- Swagger/OpenAPI 문서화

73번의 스프린트를 지나며 하나 확실해진 건 있습니다. 혼자서도 체계를 갖추면 괜찮은 서비스를 만들 수 있다는 것. 12개 에이전트와 15개 CI job과 약 2,453개 테스트가 저의 "팀"이었습니다.

Sprint 74는 아직 비어 있습니다. 다음에 뭘 할지는 모르겠지만, 이 여정이 여기서 끝나지 않을 거라는 건 알고 있습니다.
