## 제출 버튼 하나가 6개 서비스를 깨운다

사용자가 코드를 붙여넣고 "제출"을 누르면 어떤 일이 벌어질까요? 브라우저에서는 로딩 스피너가 돌아가지만, 그 뒤에서는 6개 서비스가 릴레이를 시작합니다. Gateway가 인증을 확인하고, Submission이 DB에 저장하고, RabbitMQ가 메시지를 전달하고, GitHub Worker가 코드를 push하고, AI Analysis가 Claude에게 분석을 맡기고, SSE로 결과가 브라우저까지 도착합니다.

이 글에서는 AlgoSu의 전체 아키텍처를 조감도 시점에서 살펴봅니다.

## 서비스 지도

```
┌──────────────────────────────────────────────────────┐

                   └────┬────┘
                        │
          ┌─────────────▼─────────────┐
          │   Gateway (NestJS, :3000)  │
          │   OAuth · JWT · Proxy     │
          │   Rate Limit · SSE        │
          └──┬───────┬───────┬────────┘
             │       │       │
     ┌───────▼──┐ ┌──▼────┐ ┌▼──────────┐
     │ Identity │ │Problem│ │Submission  │
     │  :3004   │ │ :3002 │ │  :3003     │
     └──────────┘ └───────┘ └──┬────┬────┘

                        ┌──────┴────▼──────┐
                        │                  │
                ┌───────▼──────┐  ┌────────▼───────┐

                │  (Node.js)   │  │  (FastAPI)     │
                └──────────────┘  └────────────────┘
```

## 각 서비스의 역할

| 서비스 | 기술 | 핵심 역할 |
|--------|------|----------|
| **Gateway** | NestJS | API 단일 진입점. OAuth 인증, JWT 발급, 서비스 프록시, SSE 스트리밍 |
| **Identity** | NestJS | 사용자·스터디·알림·피드백 관리. 34개 API 엔드포인트 |
| **Problem** | NestJS | 알고리즘 문제 CRUD, 마감 시간 스케줄러, Redis 캐싱 |
| **Submission** | NestJS | 코드 제출 관리, **Saga Orchestrator**로 분산 워크플로우 조율 |
| **GitHub Worker** | Node.js | RabbitMQ 소비자. 사용자 GitHub 저장소에 코드 push |
| **AI Analysis** | FastAPI | Claude API 코드 분석, Circuit Breaker, 일일 한도 관리 |

Gateway가 모든 외부 요청을 받아 인증을 처리한 뒤 적절한 서비스로 라우팅합니다. 각 서비스는 자신의 데이터베이스만 소유하고, 다른 서비스의 데이터가 필요하면 HTTP API를 호출합니다.

## Database per Service

AlgoSu는 3개의 PostgreSQL 데이터베이스를 분리 운영합니다.

| DB | 소유 서비스 | 주요 테이블 |
|----|-----------|------------|
| `identity_db` | Identity | users, studies, study_members, notifications, feedback |
| `submission_db` | Submission | submissions, review_comments, review_replies, drafts |
| `problem_db` | Problem | problems (별도 PostgreSQL 인스턴스) |

초기에는 Gateway가 Identity DB에 직접 접근하는 구조였습니다. 편리하지만 서비스 간 결합도가 높아져 독립 배포가 어려워졌고, [ADR-001](/posts/adr-001-gateway-identity-db-separation)에서 이를 HTTP API로 전환했습니다. "편리함"보다 "독립성"을 택한 결정이었습니다.

## 서비스 간 통신: 4가지 패턴

하나의 제출이 처리되는 과정에서 4가지 통신 패턴이 모두 등장합니다.

### 1. HTTP 프록시 (동기)

Gateway가 `http-proxy-middleware`로 요청을 내부 서비스에 전달합니다. 이때 `X-Internal-Key` 헤더를 주입하여 내부 서비스가 외부 직접 호출을 거부할 수 있게 합니다.

```
클라이언트 → Gateway → X-Internal-Key 주입 → Submission Service
```

각 서비스는 자신만의 Internal Key를 가지며, k8s Secret으로 관리합니다. `timingSafeEqual`로 비교하여 타이밍 공격을 방지합니다.

### 2. RabbitMQ (비동기 작업 큐)

즉시 응답이 필요 없는 작업은 메시지 큐로 넘깁니다. GitHub push와 AI 분석이 대표적입니다.

```
Submission ──publish──→ submission.github_push ──→ GitHub Worker
                    └─→ submission.ai_analysis ──→ AI Analysis
```

메시지는 `persistent: true`로 발행하여 RabbitMQ 재시작에도 유실되지 않습니다. 처리 실패 시 Dead Letter Queue(DLQ)로 이동합니다.

### 3. Redis Pub/Sub (실시간 상태 전파)

Worker가 작업을 완료하면 Redis 채널에 상태를 발행합니다. Gateway의 SSE 컨트롤러가 이를 구독하여 브라우저로 전달합니다.

```
AI Worker ──publish──→ Redis [submission:status:{id}] ──→ Gateway SSE
```

### 4. SSE (브라우저 Push)

브라우저는 `EventSource`로 Gateway의 SSE 엔드포인트에 연결합니다. 최대 5분간 유지되며, 30초 주기 하트비트로 연결 상태를 확인합니다. AI 분석 완료 같은 터미널 이벤트를 수신하면 자동으로 연결을 종료합니다.

## 인증 흐름

### OAuth → JWT → httpOnly Cookie

```
사용자 ──"Google 로그인"──→ Gateway
                              │
                    ① state 생성 → Redis (5분 TTL)
                    ② Google OAuth redirect
                              │
사용자 ←── Google 로그인 ────→│
                              │
                    ③ state 검증 (CSRF 방지)
                    ④ 인증 코드 → Access Token
                    ⑤ 사용자 프로필 조회
                    ⑥ Identity 서비스에 upsert
                    ⑦ JWT 발급 (HS256, 1시간)
                    ⑧ httpOnly Cookie 설정
                              │
사용자 ←── 대시보드 redirect ──┘
```

JWT는 반드시 httpOnly Cookie로만 전달합니다. `Secure`, `SameSite=Strict` 플래그를 설정하고, 알고리즘은 HS256으로 명시 고정하여 `none` 알고리즘 공격을 차단합니다. 만료 5분 이내의 토큰은 자동으로 갱신합니다.

### 내부 서비스 인증

서비스 간 통신에는 `X-Internal-Key` 헤더를 사용합니다. Gateway가 프록시 시 자동 주입하며, Traefik Ingress에서 `/internal/*` 경로를 `ipAllowList: 127.0.0.1/32`로 차단하여 외부에서 내부 API를 직접 호출할 수 없게 합니다.

## 배포 인프라

```
┌─────────────────────────────────────────────┐
│          OCI ARM (Ubuntu 24.04)             │
│                                             │
│  ┌────────────────────────────────────────┐ │
│  │            k3s (v1.34.4)               │ │
│  │                                        │ │
│  │  Traefik ─→ Gateway ─→ 내부 서비스    │ │
│  │         ─→ Frontend                    │ │
│  │         ─→ Blog                        │ │
│  │                                        │ │
│  │  PostgreSQL × 2  ·  Redis  ·  RabbitMQ │ │
│  └────────────────────────────────────────┘ │
│                    │                         │
│             Cloudflare Tunnel               │
│              (Zero Trust)                    │
└─────────────────────────────────────────────┘
         │
    algo-su.com / blog.algo-su.com
```

왜 EKS나 GKE가 아닌 k3s일까요? OCI ARM 프리티어의 제약 때문입니다. vCPU 4개, 메모리 24GB의 단일 노드에서 모든 서비스를 운영합니다. k3s는 Kubernetes의 핵심 기능을 유지하면서 메모리 풋프린트를 최소화한 경량 배포판으로, 이 환경에 적합합니다.

### GitOps 배포 흐름

```
git push → GitHub Actions CI (15 jobs)
              → Docker 이미지 빌드 (ARM64)
              → GHCR push
              → aether-gitops 레포 태그 갱신
                   → ArgoCD 감지
                   → k3s 자동 배포
```

모든 배포는 GitOps로 자동화됩니다. 소스 코드 변경 → CI → 이미지 빌드 → GitOps 레포 갱신 → ArgoCD 동기화 → k3s 배포. 수동 `kubectl apply`는 사용하지 않습니다.

### 리소스 배분

| 서비스 | 메모리 요청 | 메모리 제한 | 특징 |
|--------|-----------|-----------|------|
| Gateway | 256Mi | 400Mi | Init 없음 |
| Identity | 256Mi | 400Mi | Init: DB Migration |
| Submission | 256Mi | 400Mi | Init: DB Migration |
| Problem | 256Mi | 400Mi | Init: DB Migration |
| GitHub Worker | 128Mi | 256Mi | 메트릭: :9100 |
| AI Analysis | 512Mi | 2Gi | Claude API 호출 |
| Blog | 32Mi | 64Mi | nginx 정적 서빙 |

모든 Deployment는 `maxUnavailable: 0`, `maxSurge: 1`로 무중단 배포합니다. 컨테이너는 non-root 사용자로 실행하고, 읽기 전용 파일시스템에 모든 Linux Capability를 제거합니다.

## 마무리: 작은 팀의 현실적 MSA

AlgoSu의 아키텍처는 "교과서적 MSA"와 "현실적 제약" 사이의 타협입니다. Database per Service 원칙은 지키되 물리 인스턴스는 2개로 제한하고, 서비스 메시는 도입하지 않되 Internal Key로 인증을 보장하고, HPA 대신 고정 1 replica로 운영하되 무중단 배포는 유지합니다.
