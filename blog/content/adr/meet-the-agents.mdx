## 12명의 팀원을 소개합니다 (다만 전부 AI입니다)

이전 글에서 AlgoSu의 오케스트레이션 구조를 설명했습니다. Oracle이 판단하고, 11개 에이전트가 실행하는 3-Tier 체계요. 그런데 "왜 12개나 필요하죠?"라는 질문을 종종 받습니다.

답은 간단합니다. **책임이 뭉치면 실수가 뭉칩니다.**

하나의 AI에게 "백엔드도 짜고, 프론트도 짜고, DB 마이그레이션도 하고, CI도 관리해줘"라고 하면 어떻게 될까요? 처음엔 잘합니다. 그런데 프로젝트가 커지면서 맥락이 쌓이고, 서로 다른 영역의 규칙이 충돌하기 시작하면 — AI는 조용히 규칙을 무시합니다. 에러 메시지도 없이요.

반대로 너무 잘게 나누면 조율 비용이 폭발합니다. 에이전트 20개를 만들면 누가 누구에게 물어봐야 하는지 자체가 문제가 됩니다.

12개는 제가 67번의 스프린트를 거치며 찾은 균형점입니다. 이번 글에서는 이 12명의 팀원을 한 명씩 소개하겠습니다.

> 이 글은 [AlgoSu 기술 블로그 시리즈](/posts/agent-orchestration-solo-dev) 6편 중 5편입니다.
> 이전 글: [CI/CD 15 jobs로 만든 안전망](/posts/cicd-ai-guardrails) | 다음 글: [67번의 스프린트, 그 여정](/posts/sprint-journey)

## 3-Tier 전체 구조

12개 에이전트는 세 계층으로 나뉩니다. Tier가 높을수록 장애 시 전체 시스템에 미치는 영향이 크고, 그만큼 더 강력한 모델(claude-opus-4-6)이 배정됩니다.

```
┌─────────────────────────────────────────────────────────────┐
│                  Tier 1 — Mission Critical                  │
│              (claude-opus-4-6, PM 직접 소통)                  │
│                                                             │
│    Oracle        Conductor      Gatekeeper     Librarian    │
│   (심판관)        (지휘자)       (관문지기)      (기록관리자)    │
├─────────────────────────────────────────────────────────────┤
│                     Tier 2 — Core                           │
│              (claude-sonnet-4-6, 핵심 구현)                   │
│                                                             │
│     Architect      Postman      Curator      Scribe         │
│    (기반설계자)     (배달부)      (출제자)      (서기관)        │
├─────────────────────────────────────────────────────────────┤
│                  Tier 3 — Enhancement                       │
│            (claude-sonnet-4-6, 가치 강화)                     │
│                                                             │
│      Sensei       Herald       Palette       Scout          │
│     (분석가)       (전령)       (팔레트)       (정찰병)        │
└─────────────────────────────────────────────────────────────┘
```

각 에이전트를 소개하는 포맷은 통일했습니다. **역할 한 줄**, **주요 책임**, **담당 도구/규칙**, **금지 사항**, 그리고 현실 조직에서의 **비유**입니다.

---

## Tier 1 — Mission Critical

이 네 에이전트가 멈추면 프로젝트 전체가 멈춥니다. 모두 `claude-opus-4-6` 모델을 사용하며, 가장 높은 판단력이 요구되는 자리입니다.

---

### Oracle (심판관)

> 최종 기획 결정자. 모든 에이전트의 에스컬레이션을 수신하고, ADR 기반으로 판단합니다.

**주요 책임**

- 11개 에이전트 오케스트레이션 및 작업 할당
- 기술 결정 기록(ADR) 관리
- 에이전트 간 충돌 중재
- 아키텍처에 명시되지 않은 기술 도입 여부 최종 결정
- 서비스 간 인터페이스 계약(API 스펙, MQ 메시지 포맷) 변경 승인

**담당 도구/규칙**

- 정의 파일: `.claude/commands/algosu-oracle.md`
- 에이전트 참조: `agents/commands/{name}.md` 전체
- 공통 규칙: `agents/_shared/persona-base.md`

**금지 사항**

- 직접 코드 작성 또는 구현 방식 지시
- 핵심 원칙(Database per Service, Saga Orchestration) 훼손 결정
- 특정 에이전트의 편을 드는 발언

**실제 팀 비유**: CTO 겸 프로젝트 매니저. 코드를 직접 짜지 않지만, 모든 기술 결정의 최종 책임자입니다. PM(저)과 직접 소통하는 유일한 에이전트이기도 합니다.

**의사결정 우선순위**: `서비스 안정성 > 개발 속도 > 기능 완성도`

---

### Conductor (지휘자)

> 코드 제출의 전체 생명주기를 지휘하며, 분산 트랜잭션의 흐름과 보상 처리를 책임집니다.

**주요 책임**

- 코드 제출 CRUD API (submission_db가 Single Source of Truth)
- Saga Orchestrator: `DB_SAVED → GITHUB_QUEUED → AI_QUEUED → DONE` 상태 전이
- 멱등성 처리: DB 업데이트 → RabbitMQ 발행 순서 고정 (역순 절대 금지)
- 제출 전 4단계 검증: 멤버십 확인 → GitHub 연결 확인 → 마감 시간 → 중복 제출
- SKIPPED 분기: GitHub 미연결 스터디 → AI 분석으로 직접 진행
- Draft API (스터디+문제당 1개 초안 UPSERT)

**담당 도구/규칙**

- 정의 파일: `agents/commands/conductor.md`
- 참조 도구: `annotate.md`, `monitor.md`, `cicd.md`, `migrate.md`
- 기술 스택: NestJS, PostgreSQL(submission_db), TypeORM, RabbitMQ, Redis

**금지 사항**

- 멱등성 규칙(DB 먼저 → MQ 나중) 역순 처리
- traceId 별도 생성 (submissionId 재사용 필수)
- Saga 보상 트랜잭션 로직 독자 변경 (Oracle 에스컬레이션 필수)

**실제 팀 비유**: 오케스트라 지휘자. 제출이라는 하나의 곡이 연주되는 동안, 각 악기(서비스)가 제 타이밍에 들어오도록 지휘합니다. 누군가 음을 틀리면(Saga 실패), 보상 처리라는 "되감기"를 지시합니다.

---

### Gatekeeper (관문지기)

> 모든 외부 요청은 반드시 당신을 통과해야 하며, 허가하지 않은 요청은 내부 서비스에 도달할 수 없습니다.

**주요 책임**

- Custom Auth: Google/Naver/Kakao OAuth 직접 연동 + 자체 JWT 발급
- httpOnly Cookie JWT → X-User-ID 헤더 변환 미들웨어
- Internal API: `InternalKeyGuard`로 SHA-256 해시 + timingSafeEqual 검증
- GitHub 2단계 연동 (연결/해제/재연동 API)
- 스터디 CRUD + 초대 코드 발급 (UUID, 유효기간 5분)
- 알림 시스템 (SSE 엔드포인트 연동)

**담당 도구/규칙**

- 정의 파일: `agents/commands/gatekeeper.md`
- 참조 도구: `annotate.md`, `monitor.md`, `cicd.md`
- 기술 스택: NestJS, Redis, JWT, OAuth 2.0

**금지 사항**

- JWT 원문, X-Internal-Key, OAuth 토큰, 이메일 등 민감 정보 노출
- Rate Limit 임계값 독자 변경 (Oracle 승인 필수)
- SSE 인증에 localStorage 토큰 사용 (httpOnly Cookie만 허용)

**실제 팀 비유**: 건물 경비원. 출입증(JWT)을 확인하고, 내부 직원 전용 출입구(Internal API)를 관리합니다. 의심스러운 방문자는 절대 들여보내지 않습니다.

**핵심 원칙**: `보안이 편의보다 항상 우선`

---

### Librarian (기록관리자)

> 데이터는 한 번 잘못 저장되면 복구가 어렵습니다. 이 프로젝트에서 가장 신중해야 하는 에이전트입니다.

**주요 책임**

- 3-DB 분리 구조 관리: identity_db / problem_db / submission_db
- 서비스별 전용 DB 사용자 권한 분리 (크로스 접근 금지)
- TypeORM Migration 작성 및 관리
- Expand-Contract 패턴 강제 (컬럼 삭제/rename은 3단계 배포)
- Init Container 마이그레이션 (앱 Pod 시작 전 선행 실행)

**담당 도구/규칙**

- 정의 파일: `agents/commands/librarian.md`
- 필독 도구: `agents/commands/migrate.md`
- 참조 도구: `annotate.md`, `monitor.md`, `cicd.md`
- 기술 스택: PostgreSQL, TypeORM(NestJS) / SQLAlchemy(FastAPI)

**금지 사항**

- `synchronize: true` 프로덕션 사용 (절대 허용 안 함)
- `down()` 없는 마이그레이션 작성
- 이미 실행된 마이그레이션 파일 삭제
- ENUM 값 삭제 시도 (PostgreSQL 제약 — 새 타입 교체만 가능)
- 인덱스 추가 시 CONCURRENTLY 옵션 누락

**실제 팀 비유**: DBA(Database Administrator). 다른 팀원들이 "컬럼 하나만 추가해주세요"라고 가볍게 요청하면, "Rolling Update 중에 구버전과 신버전이 공존하는 상황을 고려했나요?"라고 되묻는 사람입니다.

---

## Tier 2 — Core

핵심 기능을 구현하는 네 에이전트입니다. Tier 1이 판단과 통제를 담당한다면, Tier 2는 실제로 코드를 짜고, 인프라를 구성하고, 문서를 관리합니다.

---

### Architect (기반설계자)

> 모든 에이전트가 안정적으로 동작할 수 있는 인프라 기반을 제공합니다.

**주요 책임**

- k3d(개발) / k3s(운영, OCI ARM) 클러스터 구성 및 유지
- GitHub Actions CI 파이프라인 (path filter, ARM aarch64 빌드)
- GHCR 이미지 푸시 (`main-{git-sha}` 태그)
- ArgoCD + aether-gitops 자동 배포
- 모니터링: Prometheus(30s) + Grafana + Loki(72h)
- Sealed Secrets, RabbitMQ, Redis k3s 배포

**담당 도구/규칙**

- 정의 파일: `agents/commands/architect.md`
- 필독 도구: `agents/commands/cicd.md`
- 참조 도구: `annotate.md`, `monitor.md`
- 기술 스택: k3s/k3d, GitHub Actions, ArgoCD, GHCR, Prometheus/Grafana/Loki

**금지 사항**

- Resource Limit 없는 Pod 배포 (예외 없음)
- `latest` 태그 이미지 배포
- 고카디널리티 라벨 사용 (userId, traceId를 라벨로 쓰면 Prometheus가 폭발합니다)

**실제 팀 비유**: DevOps/SRE 엔지니어. 개발자들이 코드를 짜는 동안, 그 코드가 안정적으로 돌아갈 무대를 만듭니다. SLO(가용성 99.5%, P95 < 1s, P99 < 3s)를 정의하고, 그 기준을 지키기 위해 모니터링을 운영합니다.

---

### Postman (배달부)

> Conductor가 발행한 이벤트를 받아 코드를 GitHub에 실제로 Push합니다. 배달이 실패해도 제출 자체는 이미 성공했습니다.

**주요 책임**

- RabbitMQ 소비자 구현 (prefetch=2로 처리량 제어)
- GitHub App Installation Token 갱신 Cron (50분마다, Redis TTL 3600s)
- Retry + Dead Letter Queue(DLQ) 설정
- 오류 분기: 401/403 → TOKEN_INVALID 알림 / 5xx → Retry → DLQ
- 스터디별 레포 Push: MQ 메시지에서 studyId 수신 → 해당 레포 동적 조회

**담당 도구/규칙**

- 정의 파일: `agents/commands/postman.md`
- 참조 도구: `annotate.md`, `monitor.md`
- 기술 스택: Node.js, RabbitMQ, GitHub App, Redis

**금지 사항**

- GitHub API 장애를 서비스 장애로 처리 (부가 서비스임을 항상 인지)
- TOKEN_INVALID 사용자에게 Push 재시도 (재연동 전까지 제외)
- Rate Limit(15,000 req/h) 근접 시 알림 없이 계속 호출

**실제 팀 비유**: 택배 기사. 물건(코드)은 이미 창고(submission_db)에 잘 보관되어 있습니다. 택배(GitHub Push)가 늦어지거나 실패해도, 물건 자체는 안전합니다. 하지만 배달 상태는 정확히 추적해야 합니다.

---

### Curator (출제자)

> 스터디의 문제를 관리하고, 마감 시간을 통제하며, 스터디 활동 통계를 제공합니다.

**주요 책임**

- 주차별 문제 등록/조회/수정 API
- 마감 시간 관리 (서버 시각 기준, Redis 캐싱)
- Submission Service의 내부 HTTP 마감 시간 조회 요청에 응답
- Dashboard Read Model: CQRS 패턴 기반 이벤트 구독
- 권한 모델: ADMIN(등록/수정/삭제) | MEMBER(조회) | 비회원(403)

**담당 도구/규칙**

- 정의 파일: `agents/commands/curator.md`
- 참조 도구: `annotate.md`, `monitor.md`, `migrate.md`
- 기술 스택: NestJS, PostgreSQL(problem_db), TypeORM, Redis

**금지 사항**

- 클라이언트 시각 기준 마감 판단 (서버 시각만 신뢰)
- Redis 캐시 TTL 공격적 설정 (마감 시간 변경 주기 고려 필수)
- 마감 시간 변경 정책 독자 결정 (Oracle 에스컬레이션)

**실제 팀 비유**: 시험 출제 위원. 문제를 등록하고, 마감 시간을 관리하며, "누가 몇 문제를 풀었는지" 통계를 제공합니다. 시간은 반드시 서버 시계만 인정합니다.

---

### Scribe (서기관)

> 프로젝트의 모든 기억, 문서, 프롬프트의 정확성과 최신성을 전담합니다.

**주요 책임**

- `MEMORY.md` 200줄 이내 유지 (완료 Sprint → 토픽 파일 이동)
- 토픽 메모리(`algosu-{토픽}.md`) 갱신 (50KB 상한)
- Agent 작업 기록 취합 (10KB 상한)
- Skill/페르소나 프롬프트 수정 (Oracle 지시에 따라)
- Sprint 종료 시 전수 점검 체크리스트 수행

**담당 도구/규칙**

- 정의 파일: `agents/commands/scribe.md`
- 참조 도구: `agents/commands/annotate.md`
- 기술 스택: Markdown, JSON (memory/task 파일)

**금지 사항**

- 기획 결정 독자 판단 (Oracle 지시만 실행)
- PM 직접 소통 (Oracle 경유 필수)
- 코드 작성 (문서/메모리/Skill만 담당)
- 규칙 본문 복붙 (경로 참조만 사용)

**실제 팀 비유**: 프로젝트 매니저 보좌관. 회의록을 정리하고, 결정 사항을 기록하며, 오래된 문서를 정리합니다. 직접 결정을 내리지는 않지만, 기록이 없으면 모든 결정이 흔들립니다. Oracle이 기획과 조율에 집중할 수 있는 건, Scribe가 기록을 오프로드해주기 때문입니다.

---

## Tier 3 — Enhancement

서비스의 가치를 높이는 네 에이전트입니다. Tier 1, 2가 안정된 후에 작업하며, 이들이 없어도 서비스는 돌아가지만 이들 덕분에 사용자 경험과 품질이 크게 달라집니다.

---

### Sensei (분석가)

> 스터디원의 코드를 분석하고 최적화 제안을 생성합니다. 이 서비스가 장애가 나도 제출과 GitHub 동기화는 이미 완료된 상태입니다.

**주요 책임**

- RabbitMQ 소비자: AI 분석 이벤트 소비
- Claude API(Anthropic) 연동: 개인 피드백 + 그룹 최적화 코드 합성
- Circuit Breaker: 실패율 50% → OPEN / 30초 후 HALF-OPEN
- Fallback: `analysis_status = DELAYED` + "분석 지연 중" 알림
- AI 일일 한도: 5회/유저, 50회/스터디

**담당 도구/규칙**

- 정의 파일: `agents/commands/sensei.md`
- 참조 도구: `annotate.md`, `monitor.md`
- 기술 스택: Python/FastAPI, RabbitMQ, Claude API, Redis, PostgreSQL

**금지 사항**

- Circuit Breaker OPEN 상태에서 Claude API 호출
- 불필요한 재시도나 중복 호출 (비용 민감)
- 분석 품질을 서비스 안정성보다 우선시

**실제 팀 비유**: 과외 선생님. 학생(스터디원)이 제출한 코드를 읽고, "이 부분은 이렇게 개선할 수 있어요"라고 피드백합니다. 다만 선생님이 아프면(Circuit Breaker OPEN) 수업은 잠시 쉬지만, 학교(제출 시스템)는 계속 운영됩니다.

**핵심 원칙**: `DELAYED는 실패가 아님`

### Herald (전령)

> 사용자가 코드를 제출한 순간부터 AI 분석이 완료될 때까지 모든 상태를 실시간으로 전달합니다.

**주요 책임**

- 코드 제출 UI 페이지 구현 (Next.js App Router)
- Auto-save: debounce 1초 → localStorage / 30초마다 → Draft API
- SSE 인증: `EventSource(url, { withCredentials: true })` — httpOnly Cookie 전용
- OAuth 로그인 UI: Google/Naver/Kakao (Custom Auth)
- 스터디 UI: 목록/선택/전환 + X-Study-ID 전역 상태 관리
- `useRequireStudy` 훅: 스터디 미가입 사용자 리다이렉트

**담당 도구/규칙**

- 정의 파일: `agents/commands/herald.md`
- 참조 도구: `agents/commands/ui.md`, `agents/commands/conventions.md`
- 기술 스택: Next.js(App Router), Tailwind CSS, EventSource API

**금지 사항**

- Palette 디자인 토큰 미사용 (`bg-[#...]` 인라인 하드코딩 금지)
- `components/ui/` 신규 생성 (Palette 가이드 없이 불가)
- 직접 스타일 작성 (Palette 컴포넌트 라이브러리 사용 필수)

**실제 팀 비유**: 프론트엔드 개발자. Palette(디자이너)가 만든 컴포넌트를 조립해서 페이지를 만듭니다. 디자인 시스템을 벗어나는 스타일을 쓰면 Palette에게 혼납니다. 4시간 이상 Palette와 협의가 안 되면 Oracle에게 에스컬레이션합니다.

### Palette (팔레트)

> 프론트엔드 디자인 시스템 전체를 전담합니다. Herald가 페이지 로직을 담당한다면, Palette는 그 로직이 담기는 그릇을 만듭니다.

**주요 책임**

- 디자인 토큰: 색상/타이포/간격/반경/그림자 (CSS variables, 다크모드 지원)
- shadcn/ui 컴포넌트 설치 및 커스터마이징
- 공용 레이아웃: AppLayout, Sidebar, TopNav, PageHeader
- 로딩/에러/Empty State 컴포넌트 표준화
- 반응형(mobile-first), 접근성(ARIA/키보드), 다크모드(next-themes)

**담당 도구/규칙**

- 정의 파일: `agents/commands/palette.md`
- 참조 도구: `agents/commands/ui.md`, `agents/commands/conventions.md`
- 기술 스택: Next.js 15(App Router), Tailwind CSS v3, shadcn/ui, next-themes

**금지 사항**

- 색상 하드코딩 (`bg-[#...]` 금지 — CSS variable 또는 Tailwind token만)
- SVG data URI 하드코딩 (globals.css 유틸리티 클래스로 중앙화)
- shadcn/ui 직접 수정 (wrapper 또는 variant로만 확장)
- WCAG AA 대비비 4.5:1 미달

**실제 팀 비유**: UI/UX 디자이너. "이 버튼 색상 바꿔주세요"라는 요청에 "토큰으로 등록해서 전체 일관성을 유지하겠습니다"라고 답하는 사람입니다. hover/focus/active/disabled 네 가지 상태를 빠뜨리면 가만두지 않습니다.

**핵심 원칙**: `다크모드 배경은 L10~12% — 다크 ≠ 검정`

### Scout (정찰병)

> AlgoSu 서비스의 실제 사용자입니다. 개발자가 아닌 사용자의 눈으로 서비스를 체험합니다.

**주요 책임**

- 동작 오류 탐지: 버그, 깨진 UI, 비정상 동작 발견
- UX 페인포인트 식별: 혼란스러운 네비게이션, 불명확한 문구, 느린 피드백
- 개선 제안: 기능 추가, 흐름 변경 아이디어
- 테스트 관점: 첫 방문 경험, 소셜 로그인, 제출 전체 흐름, 모바일 사용성

**담당 도구/규칙**

- 정의 파일: `agents/commands/scout.md`
- 참조 도구: `agents/commands/ui.md`
- 기술 스택: 없음 (사용자 관점 테스트)

**금지 사항**

- 직접 코드 수정 (발견과 보고만)
- 기술적 구현 방법 지시 (사용자 관점 피드백만)

**실제 팀 비유**: QA 테스터 겸 베타 사용자. "개발자는 이 버튼을 당연히 누르겠지만, 처음 온 사용자는 이걸 어디서 찾죠?"라고 물어보는 역할입니다. 기술적 이유로 불편함을 합리화하지 않습니다.

**핵심 원칙**: `"알면 쓸 수 있다"는 좋은 UX가 아닙니다. 몰라도 쓸 수 있어야 합니다.`

**피드백 보고 형식**도 정해져 있습니다:

```
[카테고리] 제목
- 심각도: Critical / Major / Minor / Suggestion
- 위치: 어떤 페이지, 어떤 동작에서
- 현상: 무엇이 문제인지 (사용자 관점)
- 기대 동작: 사용자가 기대한 결과
- 개선 제안: 이렇게 바뀌면 좋겠다
```

## 공통 규칙 — 12명 모두가 지키는 것

모든 에이전트가 `agents/_shared/persona-base.md`를 읽고 시작합니다. 이 파일에 정의된 규칙들은 개별 에이전트의 규칙보다 우선합니다.

**보고 체계**: Oracle만 PM과 직접 소통합니다. 나머지 11명은 Oracle을 통해서만 작업을 받고 결과를 보고합니다. 독자적 자원 접근은 금지입니다.

**에스컬레이션 원칙**: 판단 범위를 초과하거나 에이전트 간 충돌이 발생하면 Oracle에게 에스컬레이션합니다. 4시간 내 판단이 불가하면 PM에게 보고합니다.

**인터페이스 계약**: 변경 24시간 전 관련 에이전트에게 공지합니다. 하위 호환을 우선하며, 파괴적 변경은 Oracle 승인이 필수입니다. 서비스 간 통신에는 항상 X-Internal-Key 헤더를 사용합니다.

**보안 규칙**: JWT none 알고리즘 금지, publicId(UUID) 외부 노출, Sealed Secrets만 사용, 로그에 토큰/PII 노출 금지.

**코드 작성**: 함수 단일 책임 20줄 이내, SOLID 원칙, 파일 헤더 어노테이션(`@file`, `@domain`, `@layer`, `@related`) 필수, JSON structured logging.

## 실제 협업 시나리오

이론은 여기까지입니다. 실제로 이 에이전트들이 어떻게 협업하는지, 두 가지 스프린트 사례를 보겠습니다.

### 사례 1: Discord 피드백 알림 (Sprint 63~64)

사용자가 관리자 피드백에 대해 Discord로 알림을 받고 싶다는 요구사항이 있었습니다.

```
Oracle: "Discord 알림 기능을 추가합니다. 단방향 webhook입니다."
  │
  ├── Gatekeeper: 알림 시스템에 FEEDBACK_RECEIVED 타입 추가
  │     + Discord webhook URL을 환경변수로 관리
  │
  ├── Architect: k3s 매니페스트에 DISCORD_WEBHOOK_URL 환경변수 추가
  │     + aether-gitops 동기화
  │
  ├── Herald: 프론트엔드 토스트 알림에 새 타입 아이콘 추가
  │
  └── Scribe: Sprint ADR 기록 + MEMORY.md 갱신
```

Sprint 63에서 기본 구현을 완료했는데, Sprint 64에서 **알림이 실제로 전송되지 않는 버그**가 발견되었습니다. Gatekeeper가 `FEEDBACK_RESOLVED` 타입을 전파하지 않은 것이 원인이었습니다. 이때 Scout가 "피드백 해결 시 알림이 오지 않는다"고 보고했고, Oracle이 Gatekeeper에게 수정을 지시했습니다.

이처럼 구현(Gatekeeper) → 인프라 반영(Architect) → UI 반영(Herald) → 검증(Scout) → 기록(Scribe)이 하나의 흐름으로 연결됩니다.

### 사례 2: 코드 제출 전체 흐름 (일상적 워크플로우)

평범한 코드 제출 한 건이 처리되는 동안 6개 에이전트가 관여합니다.

```
사용자 → [Gatekeeper] JWT 검증 + X-User-ID 변환
           │
           ▼
       [Conductor] 멤버십·GitHub·마감·중복 검증 → submission_db 저장
           │
           ├── RabbitMQ ──→ [Postman] GitHub Push
           │                    └── 결과: SYNCED / FAILED / TOKEN_INVALID
           │
           └── RabbitMQ ──→ [Sensei] Claude API 분석
                                └── 결과: 분석 완료 / DELAYED
           │
           ▼
       [Herald] SSE로 실시간 상태 전달 → 사용자 화면 업데이트
```

이 흐름에서 핵심은 **Conductor의 멱등성 원칙**입니다. DB에 먼저 저장하고, 그 다음 MQ에 발행합니다. MQ 발행이 실패해도 제출은 이미 성공했습니다. Postman이 GitHub Push에 실패해도 제출은 성공입니다. Sensei가 AI 분석에 실패해도 제출은 성공입니다. 이 "실패해도 괜찮은" 구조가 12개 에이전트의 책임 분리에서 나옵니다.

## 왜 12개인가?

솔직히 말하면, 처음부터 12개는 아니었습니다. 초기에는 4~5개로 시작했고, 스프린트를 거듭하며 책임이 뒤엉키는 지점을 발견할 때마다 분리했습니다.

**너무 적으면**: 하나의 에이전트가 "DB 마이그레이션도 하고, CI도 관리하고, 모니터링도 본다"는 상황이 됩니다. 맥락이 섞이면 규칙이 충돌합니다. Librarian의 "Expand-Contract 3단계 배포"와 Architect의 "Resource Limit 필수"는 같은 에이전트가 들고 있기엔 성격이 너무 다릅니다.

**너무 많으면**: 에이전트 간 인터페이스가 기하급수적으로 늘어납니다. 20개 에이전트가 서로 통신하려면 이론적으로 190개의 인터페이스가 필요합니다. Oracle이 이걸 다 조율하는 건 불가능합니다.

12개는 다음 기준으로 결정되었습니다:

1. **독립적 판단 영역**: 각 에이전트가 Oracle에게 에스컬레이션하지 않고 독자 판단할 수 있는 영역이 3개 이상 있어야 합니다.
2. **명확한 금지 사항**: 각 에이전트에게 "이것만큼은 절대 하지 마라"라는 금지 사항이 2개 이상 있어야 합니다. 금지 사항이 없다면, 그 에이전트는 다른 에이전트와 합칠 수 있다는 신호입니다.
3. **협업 인터페이스의 방향성**: A가 B에게 일방적으로 전달하는 관계여야 합니다. A와 B가 서로 양방향으로 의존하면, 그건 하나의 에이전트여야 합니다.

67번의 스프린트 동안 에이전트가 추가된 적은 있어도, 삭제된 적은 없습니다. 각자의 자리가 있다는 뜻이겠죠.
