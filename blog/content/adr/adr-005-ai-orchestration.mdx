## 로딩 스피너 뒤에서 벌어지는 일

사용자가 코드를 제출하면 화면에는 로딩 스피너가 돌아갑니다. 하지만 그 뒤에서는 4단계 상태 전이가 진행됩니다. DB 저장, GitHub push, AI 분석, 결과 전달. 각 단계는 서로 다른 서비스에서 실행되고, 어느 단계에서든 실패할 수 있습니다.

이 글에서는 Saga Orchestrator 패턴으로 이 분산 워크플로우를 어떻게 조율하는지, 그리고 실패했을 때 어떻게 복구하는지를 다룹니다.

## Saga 상태 머신

제출(Submission)은 `sagaStep` 필드로 현재 상태를 추적합니다.

```
  ┌──────────┐     ┌────────────────┐     ┌────────────┐     ┌──────┐
  │ DB_SAVED │────→│ GITHUB_QUEUED  │────→│ AI_QUEUED  │────→│ DONE │
  └──────────┘     └────────────────┘     └────────────┘     └──────┘
       │                  │                     │
       │            ┌─────▼─────┐         ┌─────▼─────┐
       │            │  보상처리  │         │  보상처리  │
       │            │ (AI 계속) │         │ (DONE 전환)│
       │            └───────────┘         └───────────┘
       │
  ┌────▼─────┐
  │  FAILED  │  ← 최대 재시도(3회) 초과
  └──────────┘
```

| 상태 | 의미 | 타임아웃 |
|------|------|---------|
| `DB_SAVED` | 제출이 DB에 저장됨, GitHub push 대기 | 5분 |
| `GITHUB_QUEUED` | GitHub push 메시지가 큐에 발행됨 | 15분 |
| `AI_QUEUED` | AI 분석 메시지가 큐에 발행됨 | 30분 |
| `DONE` | 모든 처리 완료 (또는 AI 스킵) | - |
| `FAILED` | 최대 재시도 초과 | - |

각 상태 전이는 **낙관적 락**으로 보호됩니다. `WHERE sagaStep = 현재상태` 조건으로 UPDATE하여, 동시에 같은 제출을 처리하려는 시도를 차단합니다.

## 멱등성: "DB 먼저, MQ 나중"

분산 시스템에서 가장 중요한 원칙 중 하나는 멱등성입니다. AlgoSu는 **"DB 상태 변경 먼저, 메시지 발행 나중"** 순서를 엄격히 지킵니다.

```typescript
// 1단계: DB 업데이트 (낙관적 락)
const result = await submissionRepo.update(
  { id, sagaStep: SagaStep.DB_SAVED },
  { sagaStep: SagaStep.GITHUB_QUEUED }
);

// 락 획득 실패 = 이미 다른 프로세스가 처리 중
if (result.affected === 0) return;

// 2단계: 메시지 발행 (DB 성공 후에만)
await mqPublisher.publishGitHubPush({ submissionId, studyId });
```

왜 이 순서가 중요할까요? 만약 MQ 발행 후 DB 업데이트 순서라면, 서비스가 재시작되었을 때 DB에는 아직 `DB_SAVED`인데 메시지는 이미 발행된 상태가 됩니다. 타임아웃 체커가 이를 감지하면 같은 메시지를 또 발행하게 됩니다.

[ADR-002](/posts/adr-002-outbox-pattern)에서 Outbox 패턴 도입을 검토했지만, 현재 트래픽 수준에서는 낙관적 락만으로 충분하다고 판단했습니다.

## RabbitMQ 토폴로지

```
  Exchange: submission.events (topic, durable)
  │
  ├── routing: github.push
  │   └── Queue: submission.github_push (durable)
  │       └── DLQ: submission.github_push.dlq
  │
  └── routing: ai.analysis
      └── Queue: submission.ai_analysis (durable)
          └── DLQ: submission.ai_analysis.dlq

  Dead Letter Exchange: submission.events.dlx (topic)
```

메시지는 `persistent: true`로 발행하여 브로커 재시작에도 유실되지 않습니다. 발행 실패 시 지수 백오프(100ms → 200ms → 400ms, 상한 3초)로 최대 3회 재시도합니다.

**GitHub Push 메시지:**
```json
{
  "submissionId": "uuid",
  "studyId": "uuid",
  "timestamp": "2026-04-09T12:00:00Z"
}
```

**AI Analysis 메시지:**
```json
{
  "submissionId": "uuid",
  "studyId": "uuid",
  "userId": "uuid",
  "timestamp": "2026-04-09T12:00:00Z"
}
```

Worker는 `prefetch=2`로 동시 처리량을 제한합니다. OCI ARM 프리티어 환경에서 리소스를 보호하기 위한 선택입니다.

## GitHub Worker: 사용자 토큰으로 Push

GitHub Worker는 **사용자의 OAuth 토큰**으로 코드를 push합니다. 커밋이 사용자 이름으로 남아야 하기 때문입니다.

```
① 메시지 수신 (submissionId)
② 멱등성 체크 (Redis: ghw:processed:{id}, TTL 1시간)
③ 제출 데이터 조회 (Submission Service 내부 API)
④ GitHub 토큰 조회 (Gateway 내부 API)
⑤ 토큰 복호화 (AES-256-GCM)
⑥ Octokit으로 코드 push
⑦ 성공/실패 콜백 → Submission Service
```

**토큰 관리**: 사용자의 GitHub Access Token은 AES-256-GCM으로 암호화하여 Identity DB에 저장합니다. Worker는 사용 시 복호화하고, 로그에는 절대 출력하지 않습니다.

**Rate Limit 대응**: GitHub API의 `X-RateLimit-Remaining` 헤더를 감시합니다. 10 미만이면 경고 로그를 남기고, 429 응답을 받으면 `Retry-After` 헤더만큼 대기 후 1회 재시도합니다.

**토큰 무효 처리**: 401/403 응답은 토큰 만료를 의미합니다. 이 경우 GitHub push는 스킵하되, AI 분석은 계속 진행합니다. 사용자에게 SSE로 `github_token_invalid` 이벤트를 전송하여 재연동을 안내합니다.

## AI Analysis + Circuit Breaker

AI Analysis 서비스는 Claude API를 호출하여 코드를 분석합니다. 외부 API 호출이므로 장애 격리가 필수입니다.

### 분석 흐름

```
① 메시지 수신 (submissionId)
② 제출 데이터 조회 (code, language)
③ Circuit Breaker 상태 확인
④ Claude API 호출 (claude-3-5-sonnet, max_tokens=4096)
⑤ 응답 파싱 (점수, 피드백, 최적화 코드)
⑥ Submission Service 콜백
⑦ Redis Pub/Sub 상태 브로드캐스트
```

Claude에게 "알고리즘 스터디 3년차 멘토" 페르소나를 부여하고, 5가지 카테고리(정확성 30%, 효율성 25%, 가독성 15%, 코드 구조 15%, 모범 사례 15%)로 평가를 요청합니다.

### Circuit Breaker 상태 머신

```
  ┌────────┐  연속 5회 실패  ┌──────┐  30초 경과  ┌───────────┐
  │ CLOSED │───────────────→│ OPEN │────────────→│ HALF_OPEN │
  └────────┘                └──────┘             └─────┬─────┘
       ▲                        ▲                      │
       │    2회 연속 성공       │     1회 실패          │
       └────────────────────────┴──────────────────────┘
```

| 상태 | 동작 |
|------|------|
| **CLOSED** | 정상 호출. 실패 횟수 추적 |
| **OPEN** | 모든 호출 즉시 차단. 30초 대기 |
| **HALF_OPEN** | 시험 호출 허용. 2회 성공 시 CLOSED, 1회 실패 시 OPEN |

Worker가 `CircuitBreakerOpenError`를 받으면 메시지를 `NACK+requeue`합니다. 3회 재큐잉 후에도 Circuit Breaker가 열려 있으면 DLQ로 보내고, 사용자에게 "AI 분석이 지연되고 있습니다" 상태를 전달합니다.

### 일일 한도 관리

```python
key = f"ai_limit:{user_id}:{today}"
current = redis.incr(key)       # 원자적 증가
if current > DAILY_LIMIT:
    redis.decr(key)             # 롤백
    raise QuotaExceeded()
if redis.ttl(key) == -1:
    redis.expire(key, 86400)    # 첫 사용 시 24시간 TTL
```

AI 분석 실패 시에는 quota를 차감(DECR)하여 사용자에게 비용을 전가하지 않습니다.

## 보상 트랜잭션: 부분 실패를 허용하는 설계

전통적인 2PC(Two-Phase Commit)는 모든 참여자가 성공해야 전체가 성공합니다. AlgoSu는 다른 접근을 취합니다. **GitHub push와 AI 분석은 독립적으로 실패할 수 있고, 서로의 결과에 영향을 주지 않습니다.**

| 실패 상황 | 보상 처리 | 결과 |
|----------|----------|------|
| GitHub push 실패 | `githubSyncStatus=FAILED`, AI 분석 계속 진행 | DONE (push만 실패) |
| GitHub 토큰 무효 | AI 분석 스킵, 즉시 DONE | DONE (재연동 안내) |
| AI 분석 실패 | 제출은 DONE, `aiAnalysisStatus=delayed` | DONE (분석만 미완) |
| 전체 타임아웃 | 해당 단계 재시도 (최대 3회) | FAILED (3회 초과 시) |

이 설계의 핵심은 "제출 자체는 항상 완료된다"는 것입니다. GitHub push나 AI 분석은 부가 기능이지, 제출의 필수 조건이 아닙니다.

## SSE: 실시간 상태 전달

각 단계가 완료될 때마다 Redis Pub/Sub으로 상태를 발행하고, Gateway의 SSE 컨트롤러가 이를 브라우저로 전달합니다.

```
Worker ──publish──→ Redis [submission:status:{id}]
                         │
              Gateway SSE Controller (공유 subscriber)
                         │
              EventSource (브라우저)
                         │
                    UI 업데이트
```

**보안**: SSE 연결 시 httpOnly Cookie JWT로 인증하고, Submission Service에 소유권을 검증하여 다른 사용자의 제출 상태를 엿볼 수 없게 합니다(IDOR 방어).

**성능**: Redis subscriber를 공유하여 연결 풀 고갈을 방지합니다. 30초 주기 하트비트로 연결 상태를 유지하고, 최대 5분 후 자동 종료합니다.

**터미널 이벤트**: `ai_completed`, `ai_failed`, `github_token_invalid` 상태를 수신하면 SSE 연결을 종료하고, 동시에 Identity 서비스에 알림(Notification)을 생성합니다.

## 타임아웃과 재시도

Saga Orchestrator는 2분 주기로 타임아웃 체커를 실행합니다.

```
@Cron('*/2 * * * *')
async checkSagaTimeouts() {
  // DB_SAVED 상태에서 5분 경과한 제출
  // GITHUB_QUEUED 상태에서 15분 경과한 제출
  // AI_QUEUED 상태에서 30분 경과한 제출
  
  for (const submission of staleSubmissions) {
    if (submission.sagaRetryCount >= MAX_RETRIES) {
      // 최대 재시도 초과 → FAILED
      await markFailed(submission);
    } else {
      // 현재 상태에서 다시 시도
      await resumeSaga(submission);
    }
  }
}
```

재시도는 현재 상태에 따라 동작합니다. `DB_SAVED`면 GitHub push 메시지를 다시 발행하고, `GITHUB_QUEUED`면 GitHub Worker에게 다시 보내고, `AI_QUEUED`면 AI Analysis에게 다시 보냅니다. 3회 초과 시 `FAILED`로 마킹하고 수동 조사 대상이 됩니다.

## 전체 흐름 요약

```
사용자: "제출" 클릭
  │
  ▼
Gateway: JWT 검증 → Submission Service 프록시
  │
  ▼
Submission: DB 저장 (DB_SAVED)
  │ 낙관적 락 + MQ 발행
  ▼
RabbitMQ: submission.github_push
  │
  ▼
GitHub Worker: 토큰 복호화 → Octokit push → 콜백
  │
  ▼
Submission: advanceToAiQueued() → 한도 체크
  │
  ▼
RabbitMQ: submission.ai_analysis
  │
  ▼
AI Analysis: Circuit Breaker → Claude API → 콜백
  │
  ▼
Submission: advanceToDone()
  │
  ▼
Redis Pub/Sub → Gateway SSE → 브라우저
  │
  ▼
사용자: 분석 결과 확인
```

## 마무리

완벽한 분산 트랜잭션은 존재하지 않습니다. CAP 정리가 말해주듯, 분산 환경에서 일관성과 가용성을 동시에 보장할 수는 없습니다. AlgoSu는 "최종 일관성(eventual consistency)"을 택했습니다. 각 단계가 독립적으로 실패하고 복구할 수 있게 설계하고, 사용자에게는 실시간으로 진행 상황을 알려주고, 부분 실패를 허용하되 제출 자체는 항상 완료되게 했습니다.

Saga 패턴이 정답은 아닙니다. 트래픽이 늘어나면 [Outbox 패턴](/posts/adr-002-outbox-pattern)이 필요해질 수 있고, 서비스가 더 많아지면 이벤트 소싱을 고려해야 할 수도 있습니다. 하지만 지금 이 규모에서, 이 정도의 복잡도가 현실적인 타협점이라고 생각합니다.
