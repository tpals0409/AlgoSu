## 배경

AlgoSu의 Gateway 서비스가 Identity DB에 직접 접근하여 6개 엔티티(User, Study, StudyMember, StudyInvite, Notification, ShareLink)를 19개 파일에서 CRUD 수행하고 있었습니다. Identity 서비스는 마이그레이션 러너와 헬스체크만 존재하며 비즈니스 API가 0개인 상태 -- **DB per Service 원칙 위반**이었습니다.

## 결정

**전면 이관**: Identity 서비스에 비즈니스 API를 구축하고, Gateway의 모든 identity_db 직접 접근을 Identity HTTP API 호출로 전환했습니다.

| 엔티티 | Gateway 사용 파일 수 | 주요 작업 |
|--------|---------------------|----------|
| User | 7개 | OAuth upsert, 프로필, GitHub 연동, 탈퇴 |
| Study | 6개 | CRUD, 상태 검증 |
| StudyMember | 6개 | 가입/탈퇴, 권한 검증, 닉네임 |
| StudyInvite | 2개 | 초대 코드 생성/소비 |
| Notification | 3개 | CRUD, 30일 정리, SSE 발행 |
| ShareLink | 3개 | CRUD, 토큰 검증, 공개 프로필 |

## 구현 결과 (Sprint 51)

- Identity 서비스에 **34개 API 엔드포인트** 구축
- Gateway → IdentityClient HTTP 전환 완료
- DB per Service 원칙 준수 달성

## 후속

- 타 서비스(submission, problem, github-worker) → Identity 직접 호출 전환은 별도 스프린트로 진행 예정
