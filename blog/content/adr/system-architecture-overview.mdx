## 버튼 하나에 서비스 6개가 움직인다면?

사용자가 코드를 제출하고 "제출" 버튼을 누릅니다. 화면에는 로딩 스피너가 잠깐 돌고, 잠시 뒤 AI 분석 결과가 나타납니다. 사용자 입장에서는 그게 전부입니다.

하지만 그 뒤에서는 Gateway가 JWT를 검증하고, Submission Service가 DB에 저장하고, RabbitMQ를 통해 GitHub Worker가 코드를 Push하고, AI Analysis가 Claude API를 호출하고, Redis Pub/Sub으로 실시간 상태가 SSE로 전달됩니다. 하나의 제출이 완료되기까지 6개 서비스가 릴레이를 합니다.

저는 이 구조를 혼자 설계하고 운영하고 있습니다. 이 글에서는 AlgoSu의 전체 아키텍처를 조감도로 그리고, 각 서비스가 어떤 역할을 맡고, 어떻게 통신하고, 어떻게 배포되는지 기록합니다.

> 이 글은 [AlgoSu 기술 블로그 시리즈](/posts/agent-orchestration-solo-dev) 6편 중 2편입니다.

핵심 설계 원칙이 세 가지 있습니다.

**1. 멱등성 순서: DB 먼저, MQ 나중**

서비스가 재시작되면 DB에는 기록이 남아 있지만 MQ 메시지는 사라집니다. 그래서 항상 DB 상태를 먼저 갱신하고, 그 다음에 RabbitMQ 메시지를 발행합니다. 역순이면 중복 발행 위험이 생깁니다.

**2. 낙관적 락으로 역진행 방지**

모든 상태 전이에서 `WHERE sagaStep = 현재단계` 조건을 겁니다. 이미 다음 단계로 넘어갔다면 `affected === 0`이 되어 중복 처리를 막습니다.

**3. 타임아웃 + 자동 재개**

## 4가지 통신 패턴

서비스 간 통신은 용도에 따라 4가지 패턴을 사용합니다.

### 1. HTTP 동기 호출 + X-Internal-Key

Gateway에서 Identity, Submission, Problem 서비스로의 호출은 HTTP입니다. 보안을 위해 모든 내부 호출에는 `X-Internal-Key` 헤더를 붙입니다.

```typescript
// services/gateway/src/common/guards/internal-key.guard.ts

@Injectable()
export class InternalKeyGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest<Request>();
    const providedKey = request.headers['x-internal-key'];
    const expectedKey = this.configService.getOrThrow<string>('INTERNAL_API_KEY');

    if (!providedKey || typeof providedKey !== 'string') {
      throw new UnauthorizedException('Internal API Key가 필요합니다.');
    }

    if (!this.timingSafeEqual(providedKey, expectedKey)) {
      throw new UnauthorizedException('유효하지 않은 Internal API Key입니다.');
    }

    return true;
  }

  private timingSafeEqual(a: string, b: string): boolean {
    const bufA = Buffer.from(a);
    const bufB = Buffer.from(b);
    if (bufA.length !== bufB.length) return false;
    return timingSafeEqual(bufA, bufB);
  }
}
```

`crypto.timingSafeEqual`을 사용해서 타이밍 어택을 방지합니다. 키는 k8s Secret으로 관리하고, 서비스마다 별도 키를 사용합니다. 하드코딩은 절대 금지입니다.

### 2. RabbitMQ 비동기 메시징

제출 후 GitHub Push와 AI 분석은 시간이 오래 걸리는 작업입니다. 사용자를 기다리게 할 수 없으니 RabbitMQ로 비동기 처리합니다.

  <PipelineStage label="submission.github_push" detail="GitHub Worker · prefetch=2" accent={5} />
  <PipelineStage label="submission.ai_analysis" detail="AI Analysis Worker" accent={6} />

GitHub Worker의 `prefetch=2`는 의도적인 설정입니다. GitHub API의 Rate Limit과 OCI Free Tier의 제한된 리소스를 고려한 동시 처리량 제한입니다. 실패 시에는 Retry 후 DLQ(Dead Letter Queue)로 이동합니다.

### 3. Redis Pub/Sub — 실시간 상태 전파

제출 상태가 바뀔 때마다 Redis Pub/Sub으로 메시지를 발행합니다. 채널 형식은 `submission:status:{submissionId}`입니다.

### 4. SSE (Server-Sent Events) — 브라우저 실시간 스트리밍

Gateway의 SSE Controller가 Redis Pub/Sub을 구독하고, 브라우저에 `text/event-stream`으로 상태를 전달합니다.

<Pipeline>
  <PipelineStage label="Submission" detail="status 변경" accent={3} />
  <PipelineStage label="Redis Pub/Sub" detail="submission:status:{id}" accent={2} />
  <PipelineStage label="Gateway SSE" detail="공유 subscriber" accent={1} />
  <PipelineStage label="Browser" detail="text/event-stream" accent={4} />
</Pipeline>

SSE 연결에는 몇 가지 안전장치를 걸어뒀습니다.

- **최대 연결 시간 5분** — 무한 연결 방지, 타임아웃 시 클라이언트가 재연결
- **공유 Redis subscriber** — 연결마다 Redis 인스턴스를 생성하면 커넥션 풀이 고갈되므로, 단일 subscriber로 채널 리스너를 관리
- **30초 heartbeat** — 프록시가 유휴 연결을 끊지 않도록 `: heartbeat\n\n`를 주기적으로 전송
- **소유권 검증** — 제출물의 소유자만 상태를 구독할 수 있음 (IDOR 방어)

## OAuth + JWT + httpOnly Cookie 인증

인증 플로우를 한눈에 보면 이렇습니다.

<Mermaid
  caption="OAuth + JWT + httpOnly Cookie 인증 시퀀스"
  chart={`
sequenceDiagram
    autonumber
    participant B as Browser
    participant G as Gateway :3000
    participant O as OAuth Provider<br/>(Google/Naver/Kakao)
    participant I as Identity :3004

    G->>O: redirect (authorize)
    O-->>G: code
    G->>O: code → token 교환
    O-->>G: access_token
    G->>I: 사용자 조회/생성
    I-->>G: user info
    G->>G: JWT 발급 (HS256, 1h)
    G-->>B: Set-Cookie (httpOnly, Secure, SameSite)
`}
/>

몇 가지 설계 결정을 공유합니다.

**httpOnly Cookie를 선택한 이유**: localStorage에 JWT를 저장하면 XSS 공격에 취약합니다. httpOnly Cookie는 JavaScript로 접근할 수 없어서 토큰 탈취 위험이 현저히 줄어듭니다.

**자동 갱신**: TokenRefreshInterceptor가 매 요청마다 JWT 잔여 시간을 확인합니다. 만료 5분(300초) 이내면 새 토큰을 발급해서 응답 쿠키에 담습니다. 사용자는 별도의 갱신 요청 없이 자연스럽게 세션이 연장됩니다.

**알고리즘 고정**: `HS256`만 허용합니다. `algorithms: ['HS256']` 옵션을 명시해서 알고리즘 혼동 공격을 방지합니다.

CI의 첫 번째 job은 항상 `secret-scan`입니다. gitleaks로 시크릿 유출을 탐지하고, `.env` 파일 커밋도 차단합니다. 보안 스캔이 통과해야만 빌드가 진행됩니다.

### 무중단 배포 전략

모든 서비스에 동일한 RollingUpdate 전략을 적용합니다.

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0   # 기존 Pod 유지
    maxSurge: 1          # 새 Pod 1개 추가 후 전환
```

`maxUnavailable: 0`이므로 배포 중에도 항상 최소 1개의 Pod가 요청을 처리합니다. 새 Pod가 readinessProbe를 통과한 후에야 기존 Pod가 종료됩니다.

Identity, Submission, Problem 서비스는 initContainer로 DB 마이그레이션을 먼저 실행하고 앱을 시작합니다.

```yaml
initContainers:
  - name: db-migrate
    command: ["node", "./node_modules/typeorm/cli.js",
              "migration:run", "-d", "dist/src/database/data-source.js"]
```

## 리소스 배분 — 프리티어의 현실

OCI ARM 프리티어에서 운영하다 보니, 리소스 배분이 중요합니다. 각 서비스의 메모리 설정입니다.

| 서비스 | 요청(request) | 제한(limit) | 비고 |
|--------|:--------:|:--------:|------|
| Gateway | 256Mi | 400Mi | CORS, JWT, SSE, 프록시 |
| Identity | 256Mi | 400Mi | DB 마이그레이션 initContainer 별도 |
| Submission | 256Mi | 400Mi | Saga Orchestrator 상주 |
| Problem | 256Mi | 400Mi | 문제 CRUD |
| GitHub Worker | 128Mi | 256Mi | 경량 RabbitMQ 소비자 |
| AI Analysis | 512Mi | **2Gi** | Claude API 응답 파싱에 메모리 필요 |
| Frontend | 128Mi | 256Mi | Next.js standalone |
| Redis | 64Mi | 256Mi | maxmemory 200MB, LRU 정책 |
| RabbitMQ | 128Mi | 512Mi | Prometheus 플러그인 활성화 |
| PostgreSQL x2 | 128Mi x2 | 512Mi x2 | statement_timeout 200ms |

AI Analysis가 유일하게 2Gi 제한을 갖습니다. Claude API 응답(최대 8,192 토큰)을 파싱하고 JSON 구조화를 수행하는 과정에서 메모리를 많이 사용하기 때문입니다.

전체 메모리 합산은 request 기준 약 2.2Gi, limit 기준 약 5.9Gi입니다. 24GB 중 OS와 k3s 오버헤드를 빼면 충분한 마진이 있습니다.

## 보안 계층 — 기본값을 바꾸지 않는 실수 방지

모든 컨테이너에 공통으로 적용되는 보안 설정이 있습니다.

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1001
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop:
      - ALL
```

root로 실행하지 않고, 권한 상승을 차단하고, 파일 시스템은 읽기 전용으로 마운트합니다. 필요한 쓰기 경로(`/tmp`, `/var/log`)만 emptyDir로 마운트합니다.

네트워크 수준에서는 default-deny 정책으로 시작하고, 필요한 서비스 간 통신만 NetworkPolicy로 허용합니다.

## AI 분석 — Circuit Breaker와 안전한 파싱

AI Analysis 서비스는 Claude API(`claude-haiku-4-5-20251001`, `MAX_TOKENS=8192`)를 호출합니다. 외부 API 의존성이 있으니 Circuit Breaker 패턴을 적용했습니다.

API가 연속으로 실패하면 Circuit Breaker가 OPEN 상태로 전환되고, 일정 시간 동안 API 호출을 차단합니다. 이 동안 들어오는 요청은 `CircuitBreakerOpenError`를 발생시키고, RabbitMQ에서 NACK+requeue 처리됩니다.

Claude의 응답 파싱도 방어적으로 구현했습니다. LLM이 항상 완벽한 JSON을 돌려주지는 않거든요.

## 마무리 — 작은 팀의 현실적 MSA

"1인 개발인데 MSA가 필요한가요?" 라는 질문을 가끔 받습니다. 솔직히 말하면, 처음부터 6개 서비스로 시작한 건 아닙니다. 기능이 늘어나면서 자연스럽게 분리된 결과입니다.

되돌아보면 몇 가지 현실적인 교훈이 있습니다.

**DB를 일찍 분리하길 잘했습니다.** 나중에 분리하면 마이그레이션 지옥이 됩니다. 처음부터 Database per Service 원칙을 지킨 덕분에 서비스 간 결합이 느슨합니다.

**비동기 처리가 핵심입니다.** GitHub Push와 AI 분석을 동기로 처리했다면 사용자는 30초 넘게 기다려야 합니다. RabbitMQ + Saga 패턴으로 제출 즉시 응답하고, 진행 상태는 SSE로 실시간 전달합니다.

**프리티어의 한계 안에서 설계합니다.** OCI ARM 프리티어는 관대하지만 무한하지 않습니다. GitHub Worker의 prefetch=2, AI Analysis의 일일 한도 체크, Redis의 maxmemory 200MB 같은 제한은 모두 리소스를 의식한 결정입니다.

다음 글에서는 이 6개 서비스를 누가 어떻게 만들고 유지하는지 — Oracle과 11개 에이전트의 오케스트레이션 구조를 파고들어 보겠습니다.
